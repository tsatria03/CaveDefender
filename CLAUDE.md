# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

CaveDefender is an **online**, audio-only game written in **NVGT** (Non-Visual Game Toolkit, an AngelScript-based engine). All code is `.nvgt`. There is no visual rendering ‚Äî output is screen-reader speech plus HRTF spatial audio through NVGT's `sound_pool`.

It is a **client/server** game. The near-term goal is: connect to the server, log into an account, chat with other players, and walk around a shared map. Map objects (platforms, tiles, zones) are spawned through a custom game-engine DLL. The chat-center milestone is functionally complete: accounts, public/private chat, slash commands, networked player positions with locator beacons, and **spatial voice chat** (push-to-talk clips relayed through the server and played at the speaker's map position).

**Heritage note:** the folder layout and several conventions were borrowed from the **SimpleFighter** project (a separate, *offline* map-builder game), so the structure looks familiar. But CaveDefender is online and very different. Do **not** assume SimpleFighter mechanics exist here ‚Äî there is no map builder, no `.sif` map format, and no build pipeline. (CaveDefender does have enemy bots and weapons, but they are its own Wallbreaker wave system ‚Äî three modes: a **PVE** mode where you defend four walls, the reverse **EVP** mode where you attack the walls from outside while builder bots defend them, and a **PVP** mode where humans fight on both sides (one team attacks the walls, the other defends and repairs) ‚Äî not SimpleFighter's NPCs/shields. PVP was built out and is feature-complete across versions 3.1‚Äì3.2. See the [[game-mode-evp]] and [[game-mode-pvp]] memories for those designs.)

## Layout

The repo root holds three top-level folders, with **source and assets deliberately separated**:
- **`src/`** ‚Äî the code only. `src/client/` (entry `cfc.nvgt`, `includes/`) and `src/server/` (entry `cfs.nvgt`, `includes/`). **No assets live here.**
- **`cf/`** ‚Äî the runnable game's assets and launchers. `cf/client/` holds `lib/` (the engine DLL + `oggenc2.exe`), `sounds/`, `docks/`, and the `cfc.py` launcher; `cf/server/` holds `data/`, `docks/`, `lib/`, and the `cfs.py` launcher. (`cfc.py`/`cfs.py` = the two runtime launchers; the old per-side `cfcm.py`/`cfsm.py` compilers were removed ‚Äî building now goes through `build/tools.py`.)
- **`build/`** ‚Äî the unified build/release pipeline (`tools.py`, run via `tools.bat`), ported from the SimpleFighter tooling. Config in `build/tools.ini` (`[game]` name/password/`client_file`/`server_file`) plus the shared engine/7-Zip/gh paths in `~/.game_tools/tools.ini`; the version lives in `build/version.txt`. It handles git (commit/push/history/tags/reset), compiles **both** sides, packages, and creates the GitHub release. (The SimpleFighter website-update step was removed ‚Äî CaveDefender isn't published on the site.)
- **`release/`** ‚Äî compiled builds: `release/windows/CaveDefenderClient_password_is_<pw>/cfc` and `release/windows/CaveDefenderServer_password_is_<pw>/cfs`, plus the two `.7z` archives in `release/archives/`.

**How source and assets connect:** at runtime each launcher runs the `.nvgt` script from `src/<side>/` but sets the **working directory to `cf/<side>/`**, so every cwd-relative path in the code (`lib/GameEngine64.dll`, `sounds/...`, `lib/oggenc2.exe`, `docks/...`, `data/...`) resolves against `cf/<side>/`. At compile time, `build/tools.py` runs `nvgt -c` per side to make a bundle folder (NVGT auto-copies its own runtime DLLs into the bundle's `lib/`), then **copies** that side's assets from `cf/<side>/` into the bundle so they sit cwd-relative next to the exe (client: `lib` merged in ‚Äî that's where `GameEngine64.dll`/`oggenc2.exe` land ‚Äî plus `sounds`, `docks`; server: `data`, `docks`). There are **no `#pragma asset` lines** ‚Äî an earlier pragma approach was dropped because this `nvgt2` build resolves pragma asset paths relative to the *output* directory, making the `..` depth brittle; the Python copy is robust. So **no in-code path needs to change**; only the launcher cwd (runtime) and `build/tools.py` (build) know about the split.

`CLAUDE.md`, `.gitignore`, etc. sit at the repo root. The engine is the **pinned legacy NVGT fork** at `C:\nvgt2\nvgt.exe` (every launcher/compiler hard-codes it); current upstream NVGT is incompatible ‚Äî see "Non-stock NVGT engine change" and the [[engine-pinned-to-nvgt2]] memory.

## Running

There is **no test suite and no linter** ‚Äî running is scripted by the Python launchers in `cf/<side>/`, and compiling/packaging/releasing by `build/tools.py`, but it's otherwise manual, on purpose. The launchers hard-code `C:\nvgt2\nvgt.exe`; `build/tools.py` reads the engine path from `~/.game_tools/tools.ini` (pointed at the same nvgt2 build). All need Python installed, and the console ones expect `.py` associated with `python.exe`. Each launcher runs the `src/<side>` script with cwd set to its own `cf/<side>` folder so assets resolve.

- **Client:** `cf/client/cfc.py` runs `src/client/cfc.nvgt` (cwd `cf/client`). It launches `nvgt.exe` with `CREATE_NO_WINDOW` (no console) and its stdout/stderr captured to a temp log, then **watches for `COMPILE_WAIT` seconds (default 5)**: a failed compile exits fast with a non-zero code, so the launcher writes the captured output to **`errors.txt`** in the directory cfc.py was launched from (plus a popup) ‚Äî since the windowless client otherwise swallows compile errors; a clean compile keeps running, so the launcher detaches and leaves it (deleting the temp log / any stale `errors.txt`). The launcher hides its own console at startup (ctypes `ShowWindow`), so the `COMPILE_WAIT` watch runs invisibly in the background ‚Äî the client's own NVGT game window opens right away with **no lingering cmd window**, and an absent/empty `errors.txt` means "no compile errors". (Server differs ‚Äî its launcher keeps the console open, so it shows compile errors live and needs none of this.) `main()` installs the keyhook, gates on `SCREEN_READER_AVAILABLE` / `SOUND_AVAILABLE`, blocks a second instance via `gamstence.is_already_running`, loads the game-engine DLL (`load_engine()`), creates the message buffers, reads prefs, sets up the network client, then loops `mainmenu()`.
- **Server:** `cf/server/cfs.py` runs `src/server/cfs.nvgt` (cwd `cf/server`) with blocking `subprocess.run`, so its cmd window stays open the whole session (it's the server's UI). The server loads config from `data/preffs/conf.svr` and the MOTD from `data/preffs/motd.svr`, ensures the `data/` folders exist, calls `net.setup_server()`, sets a `show_game_window` title (version/port/user-limit) then hides that NVGT window with `if(!SCRIPT_COMPILED) hide_window()` (interpreted = headless, the cmd window is the UI; a compiled build keeps the window), then loops `netloop()` + `gametick()`.
- **Build a release:** `build/tools.py` (launched by `build/tools.bat`, or run with four flag args non-interactively) is the unified pipeline that replaced the old per-side `cfcm.py`/`cfsm.py`. Its **compile** step does both sides in turn: it first mirrors `build/version.txt` into both `src/<side>/includes/version.nvgt`, then runs `nvgt -c` from each `src/<side>` (bundle lands there), copies that side's assets from `cf/<side>/` into the bundle (client: `lib,sounds,docks`; server: `data,docks`), and moves it to `release/windows/CaveDefender<Side>_password_is_<pw>/<cfc|cfs>`. **Package** makes two password-protected `.7z` archives (client + server) in `release/archives/`; **release** attaches both to a GitHub release. (SimpleFighter's website-update step was removed ‚Äî CaveDefender isn't published on the site.) The engine/7-Zip/gh paths come from `~/.game_tools/tools.ini` (shared across the legacy-engine games), the per-repo config from `build/tools.ini`.

## Client / server split

Two halves of the same game:
- **Client** (`src/client/`) ‚Äî presents the UI, plays sounds, manages the local player and the message buffers, and talks to the server.
- **Server** (`src/server/`) ‚Äî holds the connected-player roster and the account store, validates logins, broadcasts chat, and runs admin commands.

They are separate codebases that share only a message protocol and an encryption key. Each has its own networking file ‚Äî both named **`net.nvgt`** (`src/client/includes/main/globals/net.nvgt` and `src/server/includes/main/globals/net.nvgt`) ‚Äî with near-identical `send()` / `get_event_message()` helpers but mirror-image `netloop()`s (the client *reacts to and presents* server messages; the server *dispatches* client requests).

## Networking

Built on NVGT's built-in `network` class (enet). The client uses `setup_client` + `net.connect`; the server uses `setup_server`. Both poll `net.request()` each frame and branch on `event.type` (`event_connect` / `event_disconnect` / `event_receive`).

- **Channels:** 0 = control/system, 1 = public chat, 2 = private messages, 3 = voice. Channels 0‚Äì2 carry the space-delimited text protocol; **channel 3 carries raw ogg voice bytes** (the client sends only the audio; the server prepends a `name x y\n` header before relaying to everyone else, and drops anything larger than `maxvoicebytes`). Received clips play through `vpool` at the speaker's position.
- **Encryption:** every packet is run through `string_encrypt` / `string_decrypt` with the key `"rscs123"`, shared by both sides in `send()` / `get_event_message()`.
- **The protocol is the contract.** Both sides agree on a set of **space-delimited** message strings (`login`, `register`, `spawn_player`, `who`, `welcome`, `disconnect`, `ping_request`/`ping_response`, `/getmotd`, `/serveruptime`, `online`/`offline`, `move`/`pos` (player position: client sends `move <x> <y>`, server broadcasts `pos <username> <x> <y> <visible-rank> <afk> <nickname>`, where visible-rank is the flag-filtered rank ‚Äî the rank only when staff with their flag on, else `player` ‚Äî afk is 1/0, and nickname is the display name, all for the P player-tracker; the roster is keyed by the **username**), `play`, `terminate`, `/motd`, `/pm`, ‚Ä¶). `online`/`offline` carry `<username> <rank> <nickname>` (display uses the nickname, the username keys the roster). Because messages are space-delimited, usernames / passwords / **nicknames** must contain **no spaces**. **When you change a message on one side, update the matching handler on the other** ‚Äî a silent client/server mismatch is the classic bug here.

## Accounts

Server-side. Each account is a **folder** at `cf/server/data/players/<username>/` (resolved cwd-relative as `data/players/<username>/` while the server runs; the folder **keeps the case the account was created with** ‚Äî `DevilGuardian`, not `devilguardian`; `account_dir` resolves it case-insensitively so login still works regardless of case) holding one small file per field, so a server host can read or edit a single value by hand: `username.usr`, `email.usr` (a **cosmetic, unverified** email collected at account creation ‚Äî the new-account form is username/email/password/confirm-password, the email must be non-empty and contain `@`, no spaces, confirm-password is a client-side match check only; stored as-is and hand-editable, **never used for login** ‚Äî but **required to exist**: `login()` refuses any account whose `email.usr` is empty/missing, so legacy accounts made before the email update must be recreated; there is no verification e-mail, though NVGT's built-in HTTP client could send one via an email API if that's ever wanted), `created.usr` (the account creation date as a human string like `July 3, 2026`, written once at registration from the engine's `DATE_*` globals; shown as "Member since" on the F6 card, `Unknown` for accounts predating the field), `nickname.usr` (the display name shown in place of the username **everywhere a name is spoken**; auto-created equal to the username, hand-editable, reverted with `/nick default`), `password.usr` (holds `argon2id$<nb_blocks>$<nb_passes>$<salt>$<hash>` ‚Äî a random CSPRNG salt and the memory-hard **Argon2id** hash of the password, cost params inline so a verify reproduces them; the salt is not secret, and there is **no SHA-256 fallback**, so any pre-Argon2 account must be re-registered), `pronoun.usr` (how third-person messages refer to the player: the bare key from `data/preffs/pronouns.svr`, e.g. `Masculine/male` / `Singular/they` / `None/unknown` ‚Äî the default is the configured they/them option, a `(name only)` entry uses the name, set in-game with `/pronoun`), `rank.usr`, `muted.usr`, `banned.usr`, and the saved-room files `pubroom.usr` (`1` = has a public room, blank = none) and `privroom.usr` (the private room's password, **stored encrypted** as a single `<ciphertext>:<key>` blob mirroring the old account `<salt>:<hash>` layout ‚Äî a fresh random per-set key (CSPRNG via `make_salt`) plus the AES-encrypted, base64-wrapped password, joined by a colon, so it's reversible and Retrieve password still works; this is **obfuscation**, the key riding beside the ciphertext keeps the password out of plain sight but isn't safe from someone who has the file ‚Äî blank = none). Saved rooms are **recreated when the host logs in** (and close on disconnect, but the files keep them for next time); a server host can hand-create `pubroom.usr` to pre-make a public room, but `privroom.usr` is encrypted, so set a private room's password from in-game instead. `muted.usr`/`banned.usr` each use one scheme: **empty = not muted/banned, `0` = indefinite/permanent, any other number = the absolute epoch it expires at** (so the old separate `muted`/`muteuntil` pair is now a single file). Field access goes through `read_field`/`write_field`; `account_exists` checks the folder; `/staff` enumerates accounts via `find_directories("data/players/*")`. `rank` is one of `player`, `moderator`, `administrator`, or `developer` (mutually exclusive, **full words only** ‚Äî no short forms). `rank_level()` maps them to `0/1/2/3` and **commands are gated by minimum level** (`/promote`/`/demote`/`/kick` ‚â• moderator; `/ban`/`/unban`/`/mute`/`/unmute`/`/motd`/`/restart`/`/fastrestart` ‚â• administrator). `is_staff` (= level ‚â• 1) is for **display only** (chat-name prefix, `/staff` listing, the "you are now‚Ä¶" notice). Both take an explicit target rank: `/promote <name> <rank>` and `/demote <name> <rank>`. You can act on anyone **at or below** your own level (yourself and peers included) but never someone above you, and a **grant cap** means you can never set a rank above your own (the anti-escalation guard). Each also enforces **direction**: `/promote`'s new rank must be *above* the target's current one, `/demote`'s must be *below* it ‚Äî so the verb always matches and you can't accidentally over-drop someone (e.g. a dev demoting to `administrator` instead of all the way to `player`). Helpers live in `src/server/includes/main/globals/account.nvgt` (`account_exists`, `create_account`, `load_account`, `read_field`, `write_field`, `verify_password`, `get_rank`, `set_rank`, `rank_level`, `is_staff`, `get_mute_until`, `set_mute`, `clear_mute`, `get_ban_until`, `set_ban`, `clear_ban`, `now_epoch`). `/mute <name> [seconds]` (0/omitted = indefinite) silences a player's chat and voice ‚Äî staff can mute **players only**, never each other; mutes persist in the account file (surviving reconnects and restarts) and timed ones auto-expire (polled in `netloop`). `/ban <name> [seconds]` (0/omitted = permanent) blocks a player entirely ‚Äî enforced both in `login()` and by kicking the target off immediately; persists the same way, and timed bans are cleared at the next login attempt (no poll needed, since a banned player is offline). `/ban` is **administrator+** with **at-or-below target protection** (can't ban above your rank; also blocks self-ban). `/kick` is **moderator+** with **at-or-below-rank protection** (you can kick players and staff at or below your own rank, including yourself and peers, but never someone above you; it's only temporary). It's the one enforcement action a moderator has ‚Äî `/warn`, `/mute`, and `/ban` are all administrator+. `/kick` and `/ban` may both target staff (at-or-below), unlike mute (players-only). `now_epoch()` returns whole Unix seconds via NVGT's Poco-backed `timestamp` (a default `timestamp` is "now"; its implicit `int64` is microseconds since 1970). Passwords are **never** stored in plaintext. The on-disk rank is loaded into the in-memory `player` roster at spawn; there is no longer a blanket `dev` backdoor (uncompiled builds used to auto-elevate ‚Äî removed); instead the configurable **owner** (`conf.svr` `owners=`, a **comma-separated list of up to 3 case-insensitive aliases for the one owner** ‚Äî the same person's alternate account names ‚Äî default `tsatria03`; the legacy singular `owner=` field is still read as a fallback) is always forced to `developer`, both on account creation and at login (`is_owner` = membership in the `owners` list, in account.nvgt), and **cannot be demoted** (the `/demote` handler refuses any owner alias, so a self-demote can't strip the owner or get reverted confusingly on next login). When an owner-alias account is **renamed** its alias is updated in the list (`replace_owner_alias`); when one is **deleted** just that alias is dropped (`remove_owner_alias`), leaving the others; both persist via `writeprefs()`. **Nicknames:** the **username is the identity** (login, folder, all command targeting); the **nickname is the display** shown in its place everywhere a name is spoken (chat, `/who`, `/staff`, the R/L/G lists, the P tracker, lifecycle/AFK notices, connect notices). `get_nickname`/`set_nickname` back `nickname.usr`; the in-memory roster carries `players[].nickname` (loaded at spawn) and ALL the `display_name*` helpers return it instead of `.name`. `/nickname`/`/nick <name>` sets your own (or `default`), `/nick <username> <altname>` lets **any staff** set an online player's (targeted by username, like `/afk`); `apply_nickname` validates (username rules; must be unique among online players' nicknames AND usernames, and must not equal **any registered account's username** other than your own ‚Äî online or offline ‚Äî so a logged-out player's identity can't be claimed as a display name), persists, and **alerts everyone** (flag+AFK-aware old name). **Command targeting:** `get_player_index_from` resolves a typed name to an ONLINE player by **username first, then nickname** (both **case-insensitive**, matching login / `account_dir` / `get_online_index` ‚Äî an account is a case-insensitive identity, so `/mute Bob` targets account `bob`), so commands accept either. Player-targeting commands now require the target to be **online** ‚Äî `/promote`, `/demote`, `/ban`, `/warn`, `/mute`, `/unmute` were converted from offline-capable to online-only and operate on `players[ti]` (persisting via `players[ti].name`); `/kick`, `/afk`, `/pm`, `/nick`, `/notifyplayer` already were. The **only** offline-capable holdout is **`/unban`** (a banned player is always offline, so online-only is impossible); it takes the username, resolved **case-insensitively** (via `account_exists` / `account_dir`) like every other targeting path. Not-found feedback is context-aware: "This player does not exist." vs "This player is not online." `/who` and `/staff` append `(username)` when it differs from the nickname.

- `register <username> <email> <password>` creates an account (the `<email>` is cosmetic ‚Äî see `email.usr` above; all three tokens are space-free); `login <username> <password> <version>` verifies it ‚Äî and the server **rejects the login if `<version>` doesn't match its own version**, so client and server must be on the same version to connect, **and rejects it if the account has no email on file** (legacy pre-email accounts). (Account creation no longer collects gender; how a player is referred to in messages is a separate **pronoun** choice, set in-game with `/pronoun` and stored in `pronoun.usr`, never at signup.)
- The client **connection menu** (`menus/menu.nvgt`) offers **sign in** (the last-used account, cached locally inside the encrypted `settings.cvf`), **sign in as** (any account, via a form), and **new account** (a form with username and masked password).
- **Pronouns** (`/pronoun`) mirror the language-channel system: `data/preffs/pronouns.svr` lists the options (one per line, `key (form/form/form/form/form)` = subject/object/possessive/possessive-pronoun/reflexive, or `none (name only)`), read fresh each time; `/pronoun` pushes a server menu (`push_pronoun_menu`, id `pronoun`), the choice persists to `pronoun.usr` and loads into `players[].pronoun` at spawn (validated, falls back to `default_pronoun()` ‚Äî the configured they/them option, found by grammar so it survives a key rename). `pronoun_word(key, form, displayname)` returns the right word for a message (name-only entries fall back to the display name, with `'s` for the possessive). Used to make third-person messages gender-aware (nickname change, staff-flag toggle, room join/leave, game invite).

## The custom game engine

Map objects are spawned by a custom engine written in **PureBasic**, shipped as `cf/client/lib/GameEngine64.dll` and wrapped in `src/client/includes/main/deps/GameEngine.nvgt` (`load_engine()`, `spawn_platform` / `spawn_zone` / `get_tile_at`, plus FTP/zip/download/recording helpers that are part of the engine's surface and not all used yet). The wrapper loads it cwd-relative as `engine.load("lib/GameEngine64.dll")`, which resolves to `cf/client/lib/` because the launcher's cwd is `cf/client` (and to the bundled `lib/` next to the exe in a compiled build). The DLL must be present, or the wrapped calls throw `"library is not loaded"`; `load_engine()` checks the load result and shows a clean alert + exits if it fails. (The voice-chat encoder `oggenc2.exe` sits beside it in `cf/client/lib/`, invoked the same cwd-relative way.)

## Non-stock NVGT engine change

This game depends on a **patched build** of NVGT (the Legacy-NVGT C++ source ‚Äî a separate repo, not this game's `src/`). `library::load()` in the engine's `src/library.cpp` was changed to resolve **any** relative path against the current working directory (not just bare filenames) before loading, so a DLL in a subfolder like `lib/GameEngine64.dll` loads correctly ‚Äî Windows' `SafeDllSearchMode` otherwise excludes the CWD. This is what lets `engine.load("lib/GameEngine64.dll")` find the DLL under the launcher's cwd (`cf/client/lib/`). This needs a `scons` rebuild of NVGT ‚Äî but it's already baked into the pinned `C:\nvgt2` fork (see [[engine-pinned-to-nvgt2]]). With **stock** NVGT the engine won't load from `lib/` at all.

## Include tree

`cfc.nvgt` includes only `includes/includes.nvgt`, which pulls in three NVGT stdlib files (`bgt_compat`, `instance`, `token_gen` ‚Äî resolved from the NVGT install, not the repo, so don't flag them as missing) then glob-includes every directory under `includes/main/`: `deps/`, `functions/`, `globals/`, `menus/`, `parsers/`. Globbing means every symbol is visible everywhere, and renamed files are picked up automatically (no explicit include to update). The **server** now mirrors this exactly: `cfs.nvgt` includes `includes/includes.nvgt`, which glob-includes `main/deps/*`, `main/functions/*`, `main/globals/*` (with `menus/` and `parsers/` stubbed out for later).

Key client files (under `src/client/includes/main/`):
- `globals/dec.nvgt` ‚Äî central client state: player identity (`name`, `password`), prefs, sound pools, timers, and the `st` savedata. The cached last account (name/password) is stored inside the same `settings.cvf` as prefs, not a separate file. (The version string now lives in `src/client/includes/version.nvgt`, not here.)
- `globals/net.nvgt` ‚Äî the connect / login / register flow and the `netloop()` that handles incoming server messages.
- `globals/game.nvgt` ‚Äî the in-chat loop: movement (which also reports position to the server via `move`), voice-chat keys, F1‚ÄìF4 server keys, F5 to toggle player beacons, the escape-to-leave handler, buffer navigation. Each frame it calls `update_sound_pools()` then `beaconloop()`.
- `globals/map.nvgt` ‚Äî the map (`spawn_map()` via the engine), stepping / wall-bounce, and the `me` / `cam` vectors and bounds.
- `globals/weapon.nvgt` ‚Äî **EVP only**: the local player's weapon (number row 1-4 = axe/bat/crowbar/hammer ‚Üí `draw_weapon`/`equip_weapon`, 3D draw cue, "x equipped" to misc) and the held-Space **swing** (`swingloop`: per-weapon windup ‚Üí `attack <name>` to the server ‚Üí recovery; doesn't lock movement, mirrors SimpleFighter's `is_swinging`). You're unarmed until a round starts.
- `globals/bot.nvgt` ‚Äî **EVP only**: the client roster of builder bots (`clientbots`, fed by `botspawn`/`botstep`/`botclear`) so the **B key** (`track_bots` in mapfuncts) can locate them, nearest-first, like P does for players.
- `globals/player.nvgt` ‚Äî the client-side roster (`players`, an array of the `player` class: `name`, `x`, `y`, a `beacontimer` and `beaconsound` handle) of **other** players the server reports, for placing their sounds in 3D. The local player stays `me` + identity globals. `netloop()` keeps it current via `add_player` / `set_player_position` / `remove_player` (keyed by `get_player_index`), skipping your own name. Also holds the beacon system: `beaconloop()` pulses `beacon.ogg` at each player's spot every `beacontime` (500 ms) on the `beaconpool`, pitch jittered 80‚Äì120, gated by `beacons_enabled` (F5).
- `globals/decpool.nvgt` ‚Äî the `all_pools` `sound_pool` array (`p`, `beaconpool`, `vpool`, `ambpool`, `musicpool`, `objpool`) and `initialize` / `update` / `pause` / `resume` helpers. `update_sound_pools()` positions the 3D listener at `me` and is now called every frame from `game()`'s loop (needed for the beacons to pan).
- `deps/buffer.nvgt` ‚Äî the message-buffer system: categories (`alerts`, `global chats`, `connections`, `private chats`, `misc`, `player events`) the player navigates (comma/period to move within a buffer, brackets to switch buffers) to review messages by type. Exported logs go under appdata `.../CaveDefender/logs`.
- `deps/` ‚Äî vendored libraries shared with SimpleFighter: `form.nvgt` (audio form), `form_menu.nvgt`, `setupmenu.nvgt`, `dlg.nvgt` / `dlgplayer.nvgt`, `sound_pool.nvgt`, `savedata.nvgt`, `speech.nvgt`, `voicechat.nvgt`, `GameEngine.nvgt`, etc. **Use `dlgmessage()` for player-facing dialogs** (the nicer wrapper), not raw `dlg()`.
- `menus/menu.nvgt` ‚Äî `mainmenu()`, the connection menu + account forms, and the preferences menu (`settingsmenu()`). (The in-game documentation menu was removed; `dockread()` remains a utility in `extrafuncts.nvgt`.)
- `functions/` ‚Äî `extrafuncts.nvgt` (helper library), `savefuncts.nvgt` (`readpreffs` / `writepreffs`, `save_last_account` / `load_last_account`), `mapfuncts.nvgt` (map/spatial helpers: `gmt`, `facing_name`, `facing_degrees`, `track_players`), and `recfuncts.nvgt` (all voice chat: `recordcheck` handles the O key like `buffercheck` handles the buffer keys, plus `prep_recording` / `prep_playback` / `transmit_voice` / `play_remote_voice` ‚Äî own voice plays centered through `vpool`, others in 3D).
- `parsers/command_parser.nvgt` ‚Äî `comparse()`, the in-chat slash-command **router** (called from `game.nvgt` on `/`). Because the game is online it does not execute commands locally (unlike SimpleFighter's `comparse()`) ‚Äî it sends chat on channel 1 and forwards each recognized command on the channel the server expects. **Rule for new commands:** validate the argument count and call `syntax_error("usage")` on bad syntax (the "Invalid command syntax. Usage: ‚Ä¶" style); route all command feedback to the `misc` buffer via `add_buffer_item`. See memory.
- `parsers/menu_parser.nvgt` ‚Äî `show_server_menu()`, which renders a **server-pushed** menu. Server-initiated, client-rendered: the server (`server/.../globals/menu.nvgt`, `broadcast_menu()`) broadcasts `showmenu <id>\n<title>\n<option>‚Ä¶` on channel 0; the client builds it with `setupmenu`/`m`, runs it (modal), and replies `menuresult <id> <choice>` (0-based option index, -1 = cancelled), which the server hands to `on_menu_result()` (a hook feature code fills in). The server draws no player UI ‚Äî it only describes the menu; only in-game clients receive it.

Server files (under `src/server/`, same `includes/main/{deps,functions,globals}` glob layout as the client):
- `cfs.nvgt` ‚Äî entry, config/MOTD load, ensure-dirs, main loop.
- `includes/main/globals/net.nvgt` ‚Äî send/receive, message dispatch, `login` / `register_account`, chat broadcast, the word `filter`, and the staff/moderation commands gated by `rank_level` (`/kick`, `/ban`, `/unban`, `/motd`, `/restart`, `/fastrestart`, `/promote`, `/demote`, `/mute`, `/unmute`, `/notify`, `/notifyplayer`; plus the public `/pm`, `/who`, `/staff`, `/help`).
- `includes/main/globals/player.nvgt` ‚Äî the **in-memory** connected-player roster (the `player` class: `name` / `peer_id` / `rank` / `version` / `x` / `y`, plus `spawn_player` / `get_player_index`), distinct from the on-disk accounts. `rank` is set from the account file at spawn. Renamed from the old `user` / `user.nvgt`.
- `includes/main/globals/account.nvgt` ‚Äî the on-disk account store.
- `includes/main/globals/game.nvgt` ‚Äî the game (cavern) spaces and round lifecycle for **all three modes** (`games[].mode` = `pve`/`evp`/`pvp`, chosen at start): phases (idle ‚Üí build ‚Üí wave), wood drops, the **host-only shared pause** (`pause_game`/`resume_game`; Open games auto-resume after 90s). **PVE** = defend ‚Äî bots attack, all-walls-down is the loss (`end_round_loss`). **EVP** = attack ‚Äî you smash the four walls while builder bots defend; its `gametick` resolution makes all-walls-down a round CLEARED (escalate, next round) and a per-round timer **or** a stall (no net wall-HP progress) the loss (`evp_clear_round`/`end_evp_loss`; score = `evp_rounds_cleared`). **PVP** = humans on both sides on one concentric arena (defenders inside, attackers outside, ring walls between): a single decisive round on the EVP-style attack/build-window clock; attackers win by downing all four walls, defenders by keeping one standing till the clock expires; a whole side emptying triggers a 45s pause‚Üíforfeit. See [[game-mode-pvp]] for the full as-built design.
- `includes/main/globals/enemy.nvgt` ‚Äî the PVE wall-smasher bots: one weapon each (1 axe / 4 bat / 2 crowbar / 3 hammer, each its own attack speed), random damage (`enemy_damage`: axe 1‚Äì3 ‚Ä¶ hammer 4‚Äì8, **also used by EVP player swings**), spawned onto the least-attacked standing wall, swinging in `enemy_attacks`, redistributing when a wall falls.
- `includes/main/globals/builder.nvgt` ‚Äî the EVP builder bots (the mirror of `enemy`): they **physically roam** the cavern, gather ground wood, and carry it to the weakest standing wall to `repair_wall` it (`builder_tick`); they broadcast `botspawn`/`botstep`/`botclear` (client bot roster + caveground footsteps), and their count escalates per round (`evp_builders_for_round`). Player wood pickup is off in EVP (the bots consume it).
- `includes/main/globals/wall.nvgt` and `globals/item.nvgt` ‚Äî server-authoritative walls (health, `damage_wall`/`repair_wall`, `border_wall`; cavern walls start at a random 50‚Äì100%) and ground wood (spawn/pickup, size-scaled cap).
- `includes/main/globals/menu.nvgt` ‚Äî the server-pushed menus (`broadcast_menu`/`send_menu`, `on_menu_result`), including the host pause menu and the in-game leave menu.
- `includes/main/functions/filterfuncts.nvgt` ‚Äî the chat filters (uses NVGT's native `regexp` via `regex()` in `extrafuncts.nvgt`; matched anywhere via a `.*(...).*` wrap since `regexp.match()` is whole-string; the old third-party `filter.dll` is gone ‚Äî see memory):
  - `filter_crash()` ‚Äî the **screen-reader crash-string filter**: hardcoded patterns that crash TTS engines (Eloquence/IBMTTS under NVDA) when spoken. **Always on, applied to every text chat channel** (global on any language incl. Unfiltered, local, and staff) ‚Äî crashing other players is an attack, not a language choice. Hardcoded on purpose; never make it skippable or move it into `chatfilter.svr` (a host could delete it).
  - `filter_chat()` ‚Äî the host **word filter** (profanity): rules in the hand-editable `data/preffs/chatfilter.svr`, read fresh each call via `load_filter_rules()` (one per line, `re=<pattern>` regex or a plain substring, `#` comments/blanks skipped). Unfiltered-exempt for global chat; local chat always filtered. The `filterstuff` switch toggles it (does not affect `filter_crash`).
  - `filterchar()` ‚Äî the **character filter** for usernames/nicknames (unrelated to the above).

## Version

The version number has a **single source of truth: `build/version.txt`** ‚Äî edit only that when opening a new changelog version block. It is mirrored into the two per-side constants `src/client/includes/version.nvgt` and `src/server/includes/version.nvgt` (each just `const string version="x.y";`, included from each side's `includes.nvgt` before the glob-includes, and read everywhere `version` is used ‚Äî the window title, the login version check, etc.) by **both the launchers and the build**: each launcher (`cfc.py` / `cfs.py`) copies `version.txt` into its side's `version.nvgt` before starting (so interpreted runs are always current), and `build/tools.py` does the same at compile time. So you never hand-edit the `version.nvgt` files; they are generated from `version.txt`.

## Audio

NVGT `sound_pool` with HRTF. Player position is the vector `me`; pools advance per frame. `cf/client/sounds/` (referenced cwd-relative in code as `sounds/...`) is organized by category, with a **per-type subfolder** holding named clips:
- `ui/` ‚Äî `buffer/`, `dlg/`, `menu/`, `misc/` (online, offline, playerchat, privchat, von, voff, pingstart, pingstop, welcome, newmotd, kick, ban, promote/demote, notify, buildprep, wavestart, wavestop, gameover, pause, resume, etc.).
- `objects/walls/<wall>/` ‚Äî e.g. `wallwood/{bump,death,hurt1-3}.ogg`; also `wallbuilding/bump.ogg` (lobby), `wallgeneric/bump.ogg` (office room), and `wallcrate/bump.ogg` (the inside/PVE spectator deck's indestructible border ‚Äî bump only, never breaks) plus `wallwoodpile/bump.ogg` (the EVP outside watcher deck's border).
- `objects/platforms/<tile>/` ‚Äî e.g. `caveground/step1-5.ogg`; also `carpet` (lobby), `marble` (office room), `gravel` (EVP outdoor exterior), and `woodpile` (the breach left when a wall is destroyed).
- `objects/items/<item>/` ‚Äî e.g. `wood` (`drop`/`loop`/`place`/`take`).
- `weapons/<weapon>/` ‚Äî `axe`, `bat`, `crowbar`, `hammer` (`draw`/`fire1-3`/`hit1-3`); the enemy bots play the `hit` clips when they strike a wall.

This game does **not** support swappable sound packs (see memory) ‚Äî paths point directly at the fixed subfolders. `map.nvgt` builds the per-type subfolder paths (`objects/walls/<wall>/bump.ogg`, `objects/platforms/<tile>/step<n>.ogg`); `spawn_map()` lays a `wallwood` border around a `caveground` interior, with `me` starting at `(1,1)` on the interior.

## Player-facing docs (docks/)

The client docks live under **`cf/client/docks/`**: `changelog.txt`, `credits.txt`, `readme.txt`, `todo list.txt`. There is no longer an in-game documentation viewer (the menu was removed), but **`changelog.txt` is still the source of truth for what shipped** and is maintained on every release.

There is **also a server-side `cf/server/docks/`**, organized into subfolders: `help/` (`player.txt`, `staff.txt` ‚Äî the help pages) and `rules/` (`player.txt`, `staff.txt` ‚Äî reserved, not wired to anything yet). The whole `docks/` tree is copied into the server build by `build/tools.py` (no pragma), so new subfolders ship automatically. Help is **server-authoritative**: `/help player` / `/help staff` (client) ‚Üí server reads `docks/help/<which>.txt` (cwd-relative under `cf/server`) and sends `helptext <which>\n<content>` ‚Üí the client shows it with `dockread(title, text)` (the text overload added so received content needs no temp file). Keep the two help files updated when commands change, since they document every command and its rank.

## Rules kept in memory (not inline, to keep this file lean)

Detailed conventions live in memory files ‚Äî follow them:
- **confirm-before-implementing** ‚Äî treat a design discussion or a question (anything ending in `?`, "what if", "I wish") as a request for a plan, **not** a green light to edit. Wait for explicit go-ahead.
- **ignore-terminal-commands** ‚Äî the user's local shell commands (the `<local-command-caveat>` / `<bash-input>` / `<command-name>` blocks, e.g. `cls`, `/compact`) are the user working their own terminal, not instructions to me; never respond to or act on them unless the user explicitly asks in their actual message.
- **list-modified-files** ‚Äî end every turn that edited files with a bare-filename "Files changed:" list.
- **no-crlf-normalization** ‚Äî don't run post-edit CRLF passes; author new file content with CRLF and let git handle it.
- **changelog-rules** ‚Äî player-facing prose, sentence/entry caps, reverse-chronological, the two-file version bump above.
- **no-sound-pack-support** ‚Äî don't add sound-pack machinery.
- **input-prompt-form-vs-dialog** ‚Äî one input field ‚Üí a virtual dialog (`vd.input_box`/`vd.password_box`); more than one field ‚Üí a real tabbable `audio_form` (like the new-account form), never a chain of sequential dialogs.
- **path-conventions** ‚Äî the full path map (appdata, sounds, server data, the DLL).

`New File.txt` in the repo root is the dev's personal scratch pad ‚Äî don't read it as documentation or write to it.
