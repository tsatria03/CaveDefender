**Server data tree** (`cf/server/data/`):
- `weapons/melee/*.mel`, `weapons/explosive/*.exp`, `weapons/archery/*.arc`
- `objects/items/wood/*.mtr`, `objects/items/ammo/*.amt`
(Server items moved from the old `data/items/` to `data/objects/items/` so they parallel the client's `sounds/objects/items/`.)

**Client sound tree** (`cf/client/sounds/`):
- `weapons/<class>/<name>/` (melee/archery/explosive)
- `objects/items/wood/<type>/`, `objects/items/ammo/<type>/`

**Extensions and what they stand for:**
- `.mel` = melee, `.exp` = explosive, `.arc` = archery (the three weapon classes)
- `.mtr` = material (wood items), `.amt` = ammunition (ammo items)

**Data format convention (SETTLED 2026-07-15).** Files are line-based `key=value` (split each line on the FIRST `=`), modeled on SF's info.sif but with CaveDefender's own additions. Three separators, each with ONE job:
- `=` separates key from value (line level).
- `,` expresses a RANDOM range as `min,max` -> random(min,max); a bare number is a fixed value. e.g. `reinforce=5,10` (random 5-10), `reinforce=7` (fixed). (SF has NO random-data syntax -- its span_int just resolves map-dimension tokens or parse_int -- so this comma-range is a CaveDefender invention; the randomization mirrors the current code's random(min,max).)
- `:` is used ONLY when a single value has genuine multi-part SUB-VALUES (e.g. `launch type=projectile:arrow` = type:subtype). Never used to separate whole settings.
Wood `.mtr` uses three plain `key=value` lines (reinforce, starter, bonus) -- no colons, since the three settings are independent. Example: `reinforce=5,10` / `starter=10,20` / `bonus=20,40` (matches today's hardcoded values). ("starter" is the canonical name for the starting-amount field on BOTH wood and ammo -- the dev's spelling, not "start".)

**Wood: 4 NAMED TYPES, decoupled from size.** oak, maple, birch, pine (real wood species, strong->weak; renamed 2026-07-15 from the earlier placeholders block/plank/drift/shaft so the names match real intuition -- oak strongest, pine weakest). The OLD three size-based names (small/medium/large_wood) are RETIRED -- the type name is now the material/kind, and its "size" (magnitude) is DATA, not implied by the name. Each wood `.mtr` sets, per type: (1) how much it REINFORCES when placed into a wall (strength per piece), (2) the amount GIVEN AT THE START (build-phase stockpile), (3) the amount given as a milestone BONUS. (These three per-type numbers replaced the old size-based hardcoded values that used to live in item.nvgt/game.nvgt.)

WOOD VALUES (SETTLED 2026-07-15; tunable in the files). Only `reinforce` differs by type (a strength gradient pine<birch<maple<oak spanning the old 5-30 span); `starter`/`bonus` are UNIFORM across ALL four types (oak included) at today's flat numbers, so no type is better to gather -- the dev flattened these deliberately, to tune later:
- oak:   reinforce=15,30 starter=10,20  bonus=20,40  (strongest)
- maple: reinforce=12,22 starter=10,20  bonus=20,40
- birch: reinforce=8,16  starter=10,20  bonus=20,40
- pine:  reinforce=5,10  starter=10,20  bonus=20,40  (weakest)
(pine's 5,10 and oak's 15,30 are exactly the retired small/large reinforce floor+ceiling, so this is a strict superset of the old tuning with two extra mid-tiers. The .mtr files and the client sound folders sounds/objects/items/wood/<type>/ were renamed to these species on 2026-07-15; the loader roster in gamedata.nvgt loads oak/maple/birch/pine.)

WOOD PLACEMENT/SPEND RULE (SETTLED 2026-07-15). Because wood is no longer ranked by size, reinforcing a wall (placing ONE piece) consumes a piece of a RANDOM type among those the player currently holds -- NOT the strongest / highest-reinforce type. The wall then gains that placed type's own `reinforce` value from its `.mtr`. So there is no "save the big pieces" micro-optimizing: grants hand you a random type, ground drops are random, and spending is random too. This REPLACES any current "pick the biggest" spend logic -- see the code-mismatch note at the bottom (globals/item.nvgt spend/give). GRANT side stays as agreed: starter and bonus each pick ONE random wood type and give that type's amount (all uniform now, so the roll only decides WHICH type lands in inventory, feeding the placement variety).

**Ammo: 4 TYPES.** arrow, gas, grenade, powder -- matching the weapon->ammo mapping (crossbow->arrow, cannon->powder, firebomb->gas, grenade->grenade; see [[ranged-weapon-architecture]]).

**Ammo `.amt` fields (SETTLED 2026-07-15).** Just TWO, the exact wood parallels: `starter` (reserve you begin a round with) and `bonus` (milestone bonus to reserve). Both randomizable (min,max). NO `pickup` field and NO `cap`: each ammo ground-item gives 1 round (exactly like a wood item gives 1 piece -- wood never had a per-pickup field), and the reserve is uncapped like the wood economy. So `.amt` is even leaner than `.mtr` (wood keeps its third field `reinforce` = the placed-piece wall strength; ammo has no place effect, since a round's effect is the weapon's damage).

GRANT RULE -- differs from wood (SETTLED 2026-07-15). Wood gives a RANDOM type + random amount for both starter and bonus (all wood is interchangeable for reinforcing, so the type is just flavor -- confirmed in code: give_wood_single picks random_wood_type(), and the milestone bonus rolls random_wood_type() too; the 3.0 changelog "a single random size rather than a mix" confirms BOTH). AMMO differs because ammo types are NOT interchangeable (arrow only feeds the crossbow, powder the cannon, gas the firebomb, grenade the grenade): so BOTH the starter and bonus ammo grants use the ammo type of the player's EQUIPPED ranged weapon (guaranteed usable), with a RANDOM amount from that type's starter/bonus range. GRANT TIMING (SETTLED 2026-07-15, Option A): the starter grant does NOT happen at round start -- at round start the client auto-arms you with the AXE (melee), so a "grant at round start" rule would never see a ranged weapon. Instead, a type's STARTER ammo is granted the FIRST time you equip a ranged weapon of that type in a round (draw the crossbow -> get its starter arrows, once per ammo type per round; draw the cannon -> its starter powder). The default weapon doesn't matter. The server tracks which ammo types it has already granted this round per player (a per-player set, reset at round start) so a re-equip doesn't re-grant. Melee equips grant nothing. The milestone BONUS ammo folds into the same rule (rides along with the first ranged equip when a bonus is pending) -- to be finalized alongside whether EVP/attack even has a milestone reward flow.

RECOMMENDED AMMO VALUES (starter / bonus, rounds; tunable in the files): arrow starter=8,12 bonus=10,15; powder starter=3,5 bonus=4,7; gas starter=3,5 bonus=4,7; grenade starter=3,5 bonus=4,7. (Crossbow/arrow gets more since it's the sustained weapon; the heavy single-shots get a handful.)

**Item sound sets (client):** wood = `drop`/`loop`/`place`/`take`; ammo = `drop`/`loop`/`take` (NO place -- you don't place ammo, taking it fills your reserve; loading is the weapon's reload). GAP: the `powder` ammo folder is currently missing `drop.ogg` (has only loop/take).

**Weapon data files** (`.mel`/`.exp`/`.arc`) follow SimpleFighter's `info.sif` key=value model. Authoritative field REFERENCE is SF's own doc: `SimpleFighter/sf/docks/builder/weapons.tp` (read it for exact semantics). CaveDefender adopts a SUBSET with SF's field NAMES; missing fields fall back to defaults; sounds are NOT listed (the engine scans the weapon folder for known clip names). See [[ranged-weapon-architecture]] for the ammo/reload/bullet mechanics that consume these values.

WEAPON FIELDS -- decided so far (being chosen one at a time, 2026-07-15; list will grow):
- `damage` -- hit damage. ALL weapons. Randomizable (min,max) or fixed. Today: axe=1,3 bat=2,4 crowbar=3,6 hammer=4,8.
- `melee` -- bool. true = swings, uses melee math, takes NO ammo; false = fires, ranged physics, CONSUMES ammo. KEPT as an EXPLICIT field (dev's call, 2026-07-15): a weapon that uses ammo must set `melee=false` to switch the ammo system ON. true for axe/bat/crowbar/hammer; false for crossbow/cannon/firebomb/grenade. (SF can default this from the category folder, but CaveDefender writes it explicitly in every file.)
- `swing speed` -- melee wind-up in ms (press -> hit lands). MELEE-ONLY, and only in MANUAL fire (fire mode 0); auto fire ignores it. Today: axe=300 bat=375 crowbar=450 hammer=600.
- `fire mode` -- 0 = manual (one shot per press, paced by `spam time`) / 1 = auto (hold to stream, paced by `repeat time`). CaveDefender adopts SF's FULL model (both modes) so a weapon can single-tap without holding space. MELEE weapons use fire mode 0 (SETTLED) -- only manual fire runs the swing-speed wind-up, so this is what keeps the weighty per-weapon wind-up AND lets you press per swing instead of holding.
- `spam time` -- manual-fire recovery ms (gates fire mode 0); runs in parallel with swing speed, effective cadence = the longer of the two.
- `repeat time` -- auto-fire recovery ms (gates fire mode 1).
- `range` -- tiles the bullet flies before expiring (ranged) / how far the swing reaches (melee). ALL weapons. SINGLE field: CaveDefender gameplay is TOPDOWN (flat x/y plane, z is audio-only for the spectator deck), so NO separate x/y/z range. Normally fixed (`range=30`); min,max allowed.
- `bullet speed` -- ms per tile of travel (lower = faster); flight time = bullet speed x distance. RANGED-ONLY (melee lands instantly via swing speed). Drives the A-plus bullet's hold-then-impact timing.
- `ammo type` -- which reserve the weapon draws/reloads from. RANGED-ONLY (melee takes no ammo). One of arrow/powder/gas/grenade. Mapping: crossbow=arrow, cannon=powder, firebomb=gas, grenade=grenade.
- `max ammo` -- MAGAZINE capacity: how many shots before a reload, and the number a reload fills back up to. RANGED-ONLY. e.g. firebomb/grenade=1 (thrown, reload each), crossbow/cannon higher. The magazine's runtime `loaded` count starts FULL (= max ammo) -- there is NO `loaded ammo` data field. SF's npc-only `ammo` field is also dropped.
AMMO MODEL (three quantities, like SF's X readout "N of M loaded, R in reserve"): (1) `max ammo` = capacity [weapon file], (2) loaded = runtime, starts full, -1 per shot, refilled to max on reload, (3) RESERVE = runtime per-player-per-type pool you reload FROM -- fed by the GATHERED ammo items; its supply numbers (`starter`/`bonus`) live in the ammo `.amt` file, NOT the weapon file, and each ground pickup adds 1 round. Loop: pick up -> reserve grows -> reload -> reserve pours into magazine (up to max) -> fire -> loaded drops -> empty -> reload.
DROPPED (not needed for CaveDefender): `launch type` (all ranged weapons are straight-flying bullets to the first wall; firebomb's ignite is driven by its burn fields, not a launch tag), `loaded ammo`, npc-only `ammo`, and z-range. SF-only fields also out: weight, unlock level, fire mode's `scadder` splash, stun mode, push x/y/z, health/life/stamina drain, reflection, refchance, cloner/glider classes.
- `status` -- a generic, EXTENSIBLE status-effect field (decided 2026-07-15, replaces the earlier "hardcode the burn" idea). One colon-delimited value packs the effect + its tuning: `status=<type>:<duration_ms>:<tick_ms>:<damage>`. FIREBOMB carries `status=burn:4000:500:3,7` (burn 4000ms, bite every 500ms, 3-7 dmg/bite). The damage sub-part uses the comma range (random 3-7); the colons separate the four sub-parts -- fully consistent with the format convention (`:` sub-values, `,` ranges), no delimiter collision (split on `:` first, then the damage on `,`). Only the firebomb has a status; cannon/grenade OMIT it (no status = impact damage only). Generic on purpose -- other statuses (freeze, poison, ...) could be added later with no code change to the field.
RELOAD TIMING (SETTLED 2026-07-15): NO reload field. The client measures the reload sound with get_sound_length and drives the timer (like swingloop's wind-up), then tells the server, which validates the reserve and moves ammo. The sound IS the timing (self-syncs). See [[ranged-weapon-architecture]].
STILL TO DECIDE: nothing outstanding on the weapon file -- the field set is complete for now; add more only if something new comes up.

## 2026-07-16 (later): preffs/ tree + the .gvr game-tuning files (BUILT: conf.gvr loader)

**conf.gvr (loader BUILT, task #16):** option-2 inheritance — `item*` lines are defaults for every item category, `wood*`/`ammo*` lines override per category. `itemcount_*`/`itemcap_*` values are `size:value` pairs matched by the map-size NUMBER (25/50/75/100); the small/medium/big/large words in keys are labels only, never parsed. `itemspawn=min,max` ms. `wallbase=` is the walls' max health (default 400); the 50-100% start roll, 2/4/8x reinforce chooser, and 4x EVP builder cap all scale off it. Loader lives in gamedata.nvgt (`item_tuning` class, `wood_tuning`/`ammo_tuning`/`wall_base_health` globals, `set_tier`/`tier_value`/`seed_tuning`/`apply_tuning_line`/`load_game_conf()`), called from cfs.nvgt after load_game_data(). Seeded with the old hardcoded values, so missing file/lines = no gameplay change. item.nvgt's wood_cap/wood_drops_per_tick now read wood_tuning; new ammo_cap/ammo_drops_per_tick twins read ammo_tuning. Ammo drops got their OWN timer (game class ammotimer/ammointerval, paused/resumed in pause_game/resume_game/freeze_world/unfreeze_world alongside woodtimer) instead of piggybacking wood's tick.

**effects.gvr (BUILT, task #17):** reshaped to two GLOBAL preset lines — `reverb=freeverb:dry:wet:roomsize:damping:width` and `muffle=lowpass:cutoff:bandwidth:mix` (the client's routing decides WHEN each applies; per-mode/space granularity deliberately dropped, could return later as reverb_pvp= overrides). Server: `effect_reverb_preset`/`effect_muffle_preset` + `load_game_effects()` in gamedata.nvgt (validates the effect-type prefix and rejects values with spaces — they ride the space-delimited wire), called from cfs.nvgt; pushed once per spawn via `fx <reverb> <muffle>` right after yourrank in BOTH spawn_player paths (fresh + reconnect restore). Client: cave_reverb_fx/cave_lowpass_fx in effect_space.nvgt are now variables (no longer const), `set_cave_fx_presets()` re-prefixes with the $reverb/$lowpass slot tags, fx handler in net.nvgt dispatch. All consumers (zones, pools, voice, burn, crosspool) read the two variables at call time, so they inherit automatically.

**Comment support:** equipment files and chatfilter.svr always skipped `#`; languages/pronouns loaders now skip `#` too (menu.nvgt). conf.svr comments survive reads but are WIPED by writeprefs() when the owner list changes; motd/peak/reports are server-written, never comment them.
