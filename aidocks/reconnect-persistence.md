The reconnect "hold-and-restore" feature (3.0's last piece, tasks #3+#4) keeps a dropped player in their room/game/watch instead of dumping them in the lobby. Built almost entirely **server-side** â€” the client already transitions into any space the server tells it to (`roomjoined`/`gamejoined`/`spectatejoined` handlers redraw), so the restore is just the server replaying the scope-entry messages.

**Grace = 45s** both sides: server `reconnect_grace_ms` (net.nvgt) and client `reconnecttime` (client net.nvgt, bumped 30000â†’45000) match so the client's auto-retry covers the whole hold.

**Hold (server player fields `held`+`held_timer`+`restoring`, player.nvgt):** On `event_disconnect`, if `has_restorable_state` (in a non-lobby space, or hosts a live room/game), `begin_hold()` instead of tearing down: broadcast `offline` (normal connections-buffer notice â€” the user chose the standard online/offline messages), despawn them from their space, and swap their dead peer id for a unique **negative sentinel** (`next_held_id`, decrementing) via `rebind_peer(old,new)` which repoints every `games[]/rooms[]` members[]/host_peer ref + the roster entry. Keep membership, host status, and inventory. A bare lobby session (nothing to restore) tears down immediately as before. **Graceful `disconnect` message (deliberate "leave server") and `/kick`+`/ban` are NOT held** â€” kick/ban remove the target from `players[]` *before* the enet drop, so `event_disconnect` sees index âˆ’1 and tears down.

**Sentinel safety:** negative peer ids never reach enet â€” the server `send()` has `if(peer_id<0) return false;` at the top (peer 0 = broadcast, positive = real). This one guard makes a held player inert to ALL outgoing loops (send_to_game, voice relay, etc.) with no per-loop change. A held player also reads as **offline everywhere**: guards on `get_online_index` (targeting/invites/is_online), `/who`, `push_player_list` (F6), the R/L/G readouts (scopemembers/roomgamemembers/allgamemembers/roomsoverview), `broadcast_pos`, `set_room_scope`'s advertise line, and `all_answered_playagain` (a held member can't stall a round reset).

**Restore (`restore_scope`, net.nvgt):** `login()` UNIFIES reconnect handling â€” for ANY name already in the roster (held OR still-live) it ADOPTS that session onto the new peer and restores, rather than the old "kick ghost, fresh lobby spawn." This kills a real race: after a client is killed, enet takes ~10-35s to notice, so a FAST manual re-login arrives while the old session is still "live" (not yet held) â€” without unification that would fresh-spawn to the lobby, not restore. Now: capture `washeld`; if not held, `net.disconnect_peer_forcefully(oldpeer)` (drop the live/ghost socket); `rebind_peer(oldâ†’new peer)`; clear `held`; set `restoring` + `restore_reannounce=washeld`; move `verified_logins`; send `done`. `spawn_player` sees `restoring` â†’ replays the player's current space (gamejoined/roomjoined/spectatejoined + phase + walls + builders + items + **preserved** wood count + positions), re-centres them (client re-centres `me` on redraw), `broadcast_pos` to re-add them â€” **no fresh spawn, no saved-room recreate, no inventory clear**. The `online` re-announce fires only if `restore_reannounce` (held â†’ was announced offline); a live takeover stays quiet (never went offline). **Mid-round phase:** a shared `send_game_phase_to(peer, gi)` (net.nvgt) sends the correct remaining time (build countdown, or an EVP wave's remaining round-timer seconds the client counts down locally) AND syncs a rest window via `waveseg 0` â€” used by restore_scope, `join_game_now`, and `do_spectate`. Before this they all sent `0` for a wave and no segment, so a mid-round arrival read "Round 1, ... 0 seconds left" and defaulted to an attack window (the bug that surfaced on the first EVP reconnect test).

**Other restore-completeness fixes** (all in `restore_scope`, since the client redraws a FRESH intact cavern on re-entry via `draw_map`): (1) **EVP weapon** â€” server player `weapon` field mirrors the client's new `setweapon <name>` notice (sent from `draw_weapon`/`equip_weapon`); restore sends `equipweapon <name>` BEFORE `gamephase` so the handler's unarmed-only auto-axe doesn't override it. (2) **Results screen** â€” server player `pending_results` holds the exact `roundover`/`evpover` string (set in `end_round_loss`/`end_evp_loss`); if you reconnect during the "over" phase unanswered, it's re-sent so you get the play-again prompt instead of being stuck AND blocking everyone's reset (an unanswered member holds up `all_answered_playagain`). (3) **Breached walls** â€” `draw_map` re-lays all four walls solid and the client enforces wall collision by tile, so restore sends a SILENT `wallbreach <dir>` (client `destroy_wall_client(dir, silent=true)` â€” re-tile to woodpile, no death boom) for each wall with `health<=0`, or you'd bump an open gap. Factored into `send_wall_breaches(peer, gi)` and ALSO called from `join_game_now`, so a mid-wave JOIN gets the breaches too. SPECTATE doesn't need it â€” a watcher roams the deck behind their own indestructible border, and the client re-tile is `!watching`-guarded anyway. This means a deliberate "leave server" (graceful `disconnect` msg) removes the roster entry, so the next login has no match â†’ genuine fresh lobby spawn â€” only unexpected drops / lingering sessions restore.

**Expiry (netloop poll):** a held session past `reconnect_grace_ms` gets the real teardown (`handle_disconnect_games`+`handle_disconnect_rooms`+remove) â€” no second `offline` (already sent at hold).

**Game teardown while held (scenarios the user asked about):** games count emptiness/host-migration by `online_member_count`/`first_live_member` (game.nvgt), not `members.length()`. If the last LIVE player leaves a started game with only held members left, `end_game` routes the held members home to `pregame_room` (so a reconnect lands in the room/lobby, never a vanished game). Host migration always picks a live member. A server **restart** wipes the in-memory hold â€” everyone returns fresh to the lobby (saved rooms recreated from disk); unavoidable without persisting live game state to disk.

This is the reconnect prerequisite PVP ([[game-mode-pvp]]) was blocked on. PVP's "pause the game during a teammate's grace, else forfeit" is a PVP-only layer ON TOP of this generic hold; co-op (PVE/EVP) games just keep running during the 45s.
