The core game of CaveDefender (beyond the finished chat-center milestone) is a **multiplayer, server-authoritative remake of the "Wallbreaker" mini-game** from the offline Sammy Center project. Reference source: `C:\Users\tonys\OneDrive\Desktop\Sammy Center\includes\` — `wallbreaker.nvgt` (main loop), `wallsmasher.nvgt` (enemies), `wood.nvgt` (resource).

**Wallbreaker reference loop (single-player):** You're in a square arena (size ~25) with four walls — left/west, right/east, north, south — each with a strength %. Enemies ("wallsmashers") spawn over time, each assigned to one wall, each with a random weapon (axe/hammer/crowbar) that sets attack speed + damage; they continuously chip that wall's strength, heard via pan (left/right walls) and volume scaled to remaining strength. Wood spawns randomly on the map; you walk over it or grab nearby (Space). You repair the wall you're standing at (Enter) for +0-3 strength, costing 1 wood. If any wall drops below 0, the stronghold crumbles — game over, score = seconds survived. Keys: arrows move, C coords, W wood count, S wall strengths.

**How it maps to CaveDefender:** the existing movement/map/net/voice layer is the world layer. The sounds already match: `objects/walls/wallwood/{hurt1-3,death}` = walls taking damage/breaking, `player/items/wood/{drop,take,place,loop}` = wood spawn/pickup/place, `player/weapons/{axe,crowbar,hammer}/{draw,fire,hit}` = weapons (in the reference these belong to the ENEMY wallsmashers — whether players also wield them in CaveDefender is an open design question), `objects/platforms/cave/step1-5` = floor.

**Decided so far:**
- **Current scope = PvB (player vs. bots).** For now the only adversaries are the bot wallsmashers; players cooperate against them. This is the starting mode and will be expanded later (e.g. player combat, PvP, more modes) as the game grows — design with that expansion in mind.
- **Server-authoritative**: server owns walls, bots, wood spawns, inventories, round/lose state; clients send intents (move/grab/repair) and render audio.
- **Co-op shared stronghold**: everyone on the map piles wood onto the same four walls to strengthen them; bots smash them. Player combat (wielding the axe/hammer/crowbar themselves) and multiple game modes come LATER — first build is build/repair-only co-op vs. bots.
- **Rooms system** layered on the lobby: lobby (social) -> room (a sub-lobby / "office") -> game instance (defense round) runs inside the room. A room host can start/stop games, kick, set/retrieve password, delete the room.
- **Private rooms & passwords (LOCKED):** both public and private rooms are **listed** (public in "Join public room", private in "Join private room", shown by name). Public = open, no password, no Set/Retrieve options. Private = password-gated and **always has a password from creation**: choosing "Create a private room" immediately prompts for a **required, non-empty** password (masked input; cancel = room not created, empty = re-prompt). Joining a private room = pick it from the list, then enter its password. The room menu's **Set password** (change it) and **Retrieve password** (recall to share) appear **only for private rooms**. Password entry is a masked input box, not a form — consistent with the menus-not-forms rule (a single input is fine, like account creation).
- **Rooms have no custom names (LOCKED):** a room is identified by its **host** — displayed as "<host>'s room". No name input on create (avoids name collisions/validation/space issues; host names are already unique + space-free). The join-room list is self-describing, e.g. "Bob's room, N players, M games in progress" (exact wording TBD when the join menu is built). Consequence: **up to one public AND one private room per host (two max)** — creation is blocked only per type (`hosts_type(peer, priv)` on the server). **Rooms are PERSISTENT (LOCKED):** "Leave room" just returns you to the lobby — the room stays open (the host owns it even while stepped out, so others can join / they can return). A room closes only when its host **Deletes** it or **logs off the server** (host disconnect closes it and returns any remaining members to the lobby). No host migration for rooms (unlike open games). The per-type host limit is enforced via the host still owning the room while in the lobby (`hosts_type(peer, priv)`); disconnect closes ALL of a host's rooms.
- **Room vs. game cap (LOCKED):** a **room holds UNLIMITED players** (it's just another lobby/office, bounded only by the server's overall userlimit). A **game instance caps at 4 players** — the thematic reason is the map has exactly **4 walls** (up to one defender per wall). Many people can be in a room; they form up-to-4-player games inside it. (This replaces the earlier "room = 1-4 players" idea, which is no longer the model.)
- **UI is MENU-based, never forms.** Both menus and forms are CLIENT-rendered (the server has no player-facing UI and never renders either); the server only supplies dynamic data (room list, member list) and processes the chosen actions. Lobby menu, room menu, and the start-game map-size picker are all menus. Forms were ruled out for the game UI. **Clarification:** "no forms" means no multi-field config forms — single-input dialogs via `virtual_dialogs` (e.g. `vd.input_box`, masked password prompts) ARE allowed; they ride on the `audio_form` object as a backend, which is fine. So the chat input box (already used in `game()`) and password prompts are OK.
- **Map size chosen via a menu** when the host hits Start game: Small 25x25, Medium 50x50, Large 100x100 (grid sizes). Maps are **hardcoded** (no server maps/ folder, no map files); `spawn_map()` will take a size and lay the wall border + cave interior to that grid. (Dead map scaffolding in client `map.nvgt` — `mapdata`/`mapname`/`mapper`/`mpool`, the global `minx..maxz` bounds, `tileslot`/`wallslot`, `wall_hit_once`, and the unused `runtimer`/`walktimer` — was already removed.)
- **Difficulty scaling, two dials:** map size raises wood count, build/collection-phase length, and travel difficulty (walls farther apart); player count raises bot count/speed (so a full room of 4 isn't trivial). Both feed bot pressure. Strawman: Small ~30s build / x1 bots, Medium ~60s / x1.5, Large ~90s / x2, then multiplied by player count. Numbers to tune.
- **Round flow:** build/prep phase FIRST (timed, host may be able to skip), then waves of bots.
- **Games within a room (current model):** a room (unlimited people) hosts multiple independent game instances, each up to 4 players. Room menu game options: **Start game**, **Join game** (lists only joinable open games with a free wall/slot; refuses solo games with a "single-player game" message), **Stop game**.
- **Stop game permission (LOCKED):** regular players can stop only their own game (the one they host); **stopping any game is staff-only (moderator+)** — the "stop any game" list is a moderation tool, to prevent griefing.
- **Two access points for stopping (LOCKED):** you stop/leave YOUR OWN game from *inside* it via an **in-game Escape menu** (Resume / Leave game / Stop game [host only]) — same Escape pattern the current `game()` loop uses. The **room menu's "Stop game" list** is for stopping games you're NOT in = the staff (moderator+) tool (also lets a host who left a game running kill it from the room). This resolves "how do you stop your own game if you're in it" — you don't use the room menu for that.
- **Leave vs. stop, and host leaving (LOCKED):** "Leave game" exits you to the room while the game continues for others; "Stop game" (host) ends it for everyone. For a **solo** game, leaving = ending (you're the only one). For an **open** game, when the host picks Leave the **host migrates** to another player still inside and the game continues; it ends only when the **last** player leaves (empty).

- **Game capacity (LOCKED):** **Solo vs. Open.** When starting a game (after the map-size menu), one menu: Solo (private, unjoinable) or Open (anyone in the room joins up to 4 / a free wall). No exact 2/3/4 cap.
- **Mid-game join (LOCKED):** **build-phase-only.** You can join an open game until the bots arrive; if you arrive after the build phase, you wait in the room and join the next round's build phase. (One rule — "wait for next round" is just the automatic consequence of missing the build window.)
- **Round end (LOCKED):** **endless survival** for now — walls crumble -> collective game over, score = seconds survived -> back to room idle, host can restart. Wave-based-with-a-win is deferred to a later **mode** (the modes system swaps the end condition, not the whole game).

**Pause model (LOCKED, built in 6d when there's a sim to freeze):** pausing is a real concept only once bots/timers exist. **Solo:** the host (= you) can truly pause — server freezes that one-player game's simulation + local audio pause. **Open (multiplayer):** pause is **HOST-ONLY and rank-independent** (the game's host controls it, NOT server staff rank — a mod/dev who merely joined can't pause someone else's game). Dev's reason: stop a non-host jerk from pause-spamming. A host pause is a SHARED pause — server freezes the game and pauses it for everyone in that game (broadcast via the existing server-pushed menu system: `show_server_menu`/`broadcast_menu`, `showmenu`/`menuresult` in menu_parser.nvgt), and **only the host can Resume**. In-game pause menu "Stop game" shows for the HOST OR any STAFF (moderator+) inside the game (dev add) — so staff can stop a game from inside it, not just from the room list; server `stopgame` (no-arg) allows host-or-staff. Non-host Escape in an open game (while NOT paused) = a personal Leave-only menu that does NOT pause the shared game and does NOT pause their audio (so they stay aware while the round runs). **During a host pause: non-hosts CANNOT pause/resume but CAN ALWAYS LEAVE (dev revised the earlier "no menus during pause" idea).** A non-host pressing Escape — even while the host has the game paused — gets the same "Are you sure you want to leave the game?" Yes/No confirmation solo uses, and may leave at ANY time. So a host pause never traps anyone; the host controls only the freeze, not whether others may bail. Backstops against an AFK/abusive host: (1) host-disconnect during a pause migrates the host (or ends an empty game); (2) a server pause TIME LIMIT — if the host doesn't resume within the cap, the game auto-resumes AND the host role passes to the next player in line; (3) host-migration target = the next-in-line successor by JOIN ORDER (CONFIRMED by dev: the player who entered right after the host = `members[0]` after the host is removed, which is what `leave_game`/`handle_disconnect_games` already do). The auto-resume timer mainly serves players who'd rather keep playing than leave (since anyone can already bail). **Pause timeout is a PER-GAME setting the host picks at Start (CONFIRMED):** a menu AFTER the map-size picker offering 30 / 60 / 90 / 120 seconds (labels in seconds, not minutes — 90/120 don't read cleanly as minutes). This adds a 3rd field to the `startgame` message (`startgame <size> <mode> <pausetimeout>`) stored on the game object. To be wired WITH the pause feature in 6d (not now — until pause exists it'd be a dead menu in an empty cavern); decision is locked. The local `pause_pools()` shared-pause built in 6a Pass 1 is the SOLO/local piece; the open-game shared pause is the 6d addition. The local menu system alone can't show a menu to other players — the server-pushed menu mechanism is the tool. **6d DONE (2.0) — tasks #10-17, all built (PVE wave/bots shipped).** LOCKED decisions (designed via Q&A, differ from Sammy in places):
- **Bots (wall-smashers):** spawn only during the WAVE phase; spawn cadence SCALES WITH PLAYER COUNT (solo ~ Sammy's 10-25s baseline, more players = more/faster). Each new bot joins the wall with the FEWEST current attackers (EVEN SPREAD, not random — dev's call: random clumping wastes time scrambling on big maps). Bots are UNKILLABLE (no player weapons yet) and never despawn on their own; a bot leaves a destroyed wall and redistributes to a standing one.
- **Weapons (own speed+power, heavier = slower + stronger, real-world logic):** axe every 0.5s dmg 1-3; crowbar every 1.0s dmg 3-6 (uses random `weapons/crowbar/hit1-3.ogg`); hammer every 1.5s dmg 5-10 (`weapons/hammer/hit.ogg`); axe uses `weapons/axe/hit.ogg`. Enemies use ONLY the `hit` sounds (draw/fire are for future player weapons). Damage is much higher than Sammy's 0-3 (dev wanted harder).
- **Hit audio:** REAL 3D positioned at the wall (across the wall span, like the existing `play_wall_hurt`), volume scaled to the wall's remaining strength (weaker = louder/frantic). NOT Sammy's fake panning (Sammy used `sound.pan` -100/0/+100 with N/S both centered — can't tell N from S; we have HRTF so we don't need that hack).
- **Wall destruction (PERMANENT):** a wall at 0 is destroyed for good — re-tiled to a RANDOM `debris`/`woodpile`/`rubble` floor (walkable, own step sounds, so you HEAR a breach), can't be reinforced again, no longer attacked. This REPLACES the current client wallcheck re-tile-to-cave + the earlier "rebuildable" idea. Add a MAP-EDGE CLAMP so you can stand in a breach but can't walk off the map (bots are positionless/abstract, nothing comes through).
- **Lose condition:** game over when ALL FOUR walls are destroyed (not the first — dev's fairness call; bots-leave-dead-walls prevents the bot-sink exploit and makes the survivors fall faster = natural death spiral). Score = SECONDS SURVIVED FROM WAVE START (build/prep doesn't count), announced to everyone in the game at game over; then back to idle, host restarts. No persistent leaderboard yet (added to todo list as a later feature).
- **Pause = HOST-ONLY SHARED PAUSE:** host pauses freeze the whole sim (bots/timers) for everyone. Solo = pause INDEFINITELY; Open = auto-resume after 90s. Non-hosts can always Leave during a pause, other input frozen. Host leaves/disconnects (paused or not) -> host migrates to next player, game continues (resumes if paused). Server-driven.
- Walls START at random 20-80% of max (6c) and reinforcing has NO cap (6c) — both still apply to standing walls.

**Server-authoritative menu conversion (in progress, 1.9):** dev wants EVERY in-game menu handled server-side so menu actions ride reliable sends and survive a brief server outage (like chat does), instead of the old client request-and-wait-3s-then-give-up. RULE: lists/choices = server-pushed menus (`showmenu`/`menuresult`); typed text = client-side local box that sends a reliable command the server validates (server `ask_input(peer, prompt, prefix, masked)` -> client shows box -> sends "<prefix> <text>" -> existing command handler acts). Local-only menus (preferences, sign-in/connection) STAY client-side. Server menus cancel with escape (no Back item; dev OK'd dropping Back). Infra: client `show_server_menu`/`menu_parser.nvgt`, server `broadcast_menu` (all) + new `send_menu(peer,...)` (unicast) + `on_menu_result(index,id,choice)` in menu.nvgt; per-player `menu_targets` (int[]) maps a returned list-choice index back to a room/game id. **ALL in-game menus converted & confirmed working (lobby, room, in-game).** Client `openmenu` (Enter; server picks lobby/room/game menu by the player's room_id) + `openpausemenu` (Escape in a game). Server builders in menu.nvgt: `push_room_menu`, `push_game_list("join"/"stop")`, `push_kick_list`, `push_start_mode`/`push_start_size` (mode rides in `menu_pending`), `push_game_menu` (round controls + status line), `push_pause_menu`; `confirm_menu(peer, command, title)` (id "confirm:<cmd>"; client recognises the prefix and says "canceled" on No/escape). Per-player on the server: `menu_targets` (string[], the arg per list option — room id/game id/player name), `menu_actions` (string[], action key per option of a heterogeneous menu), `menu_pending`. Two relay directives the client handles: **`runcommand <cmd>`** (no-input choices — client just re-sends `<cmd>`, reusing the existing command handler) and **`askinput\n<prompt>\n<prefix>\n<mask>`** (typed text — client shows a local box and sends "<prefix> <text>"). So lists/choices are server-pushed; text entry stays a local box; every actual action still flows through the existing createroom/joinroom/kickfromroom/deleteroom/setroompassword/getroompassword/startgame/joingame/stopgame/beginround/skipbuild/leavegame/leaveroom handlers. `do_create_room`/`do_join_room` extracted in room.nvgt. The pause menu's LOCAL audio pause lives in client `show_server_menu` (pause/resume pools while the "pause" menu is open). REMOVED dormant client menus: `lobby_menu()`,`join_room_menu()`,`room_menu()`,`game_menu()`,`pause_menu()` + their helper flows; `room_menu.nvgt` now holds ONLY the R/L/G glance helpers (those are spoken announcements, still client request-reply, NOT converted — not menus). Server menus have no "Back" item (escape cancels; dev OK'd). Only LOCAL menus stay client-side: preferences/settings and the pre-connection sign-in/new-account menus.
