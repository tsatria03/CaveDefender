Rules for writing new entries in `docks/changelog.txt` (flat file at the docks root — CaveDefender is NOT a map builder, so there is no `docks/main/` or `docks/builder/` split like SimpleFighter had):

- **A changelog is a RECORD OF WHAT CHANGED, not a manual.** The readme is the manual and the help pages are the reference — the changelog just says, in a sentence or two, what's new or different. If an entry starts explaining *how to use* a feature step by step (keys, sub-options, mechanics), it has drifted into readme territory; pull it back to the high-level "what changed" and let the readme/help carry the how. (The [[todo-list-one-sentence]] file follows the same spirit: it tracks what's done and what's left, one short sentence each — not a description of how anything works.)
- **A whole feature that lands in ONE version gets ONE concise "added X" entry plus a readme note — NOT one entry per sub-feature/mechanic.** Learned 2026-07 on Free Play ([[game-mode-free]]): I wrote four detailed entries (mode-added, arena menu, cavern controls, self-healing walls) and the dev deleted all but the mode-added line + the "updated the readme" line, because the arena/controls/self-heal ones spelled out keys, menus, and step-by-step mechanics — readme material. When a big feature ships whole in a single version (changelog written at the end, not staged), say "Added a new game mode, free play, a sandbox..." in one entry describing what it IS, add the readme note, and let the readme section carry every mechanic. This is the single-version case; the multi-version staging below is different (each version's chunk earns its own entry as it lands).
- **Player-facing prose only.** Describe what the player sees, hears, or can now do — never the code-side cause, internal field names, parser branches, or refactors. If a change has no observable effect for the player, it does not belong in the changelog at all. (The existing file does include a few server-optimization lines, but default to player-observable wording.)
  - **"Updated the todo list / readme" meta-entries ARE valid player-facing entries — do NOT flag them for removal.** Confirmed by the dev 2026-07: players actively read the client docks (the todo list and readme in `docks/`), so a boilerplate line like "Updated the todo list file with the latest bug fix and feature improvements" is legitimately player-facing (it tells readers those docs changed). Recurring convention across versions (e.g. 4.0, 4.5). Only the truly invisible dev/code stuff (loaders, refactors, internal fields) is excluded.
- **One entry = one line of one or more sentences.** Min 1 sentence, max 3. **Default to 1.** Use 2 only for a meaningful caveat, a why, or a paired side-effect the player should know. Use 3 only when genuinely substantial. Don't pad to fill a sentence count.
- **No breathless run-ons — write each entry clean the FIRST time.** Each sentence carries ONE clear idea; never stack many clauses or a semicolon-joined pile-up into a single sentence "spoken in one breath." Prefer a period over a semicolon. When a change has many facts, split them across the 1–3 sentences, or push the mechanical detail into the readme/help and keep the changelog high-level — never cram it all into one long sentence. (The dev has to read every entry by ear via screen reader, so a run-on is genuinely painful; a whole major version of them becomes a big cleanup. Writing it right up front avoids that. Learned 2026-07 after cleaning up run-ons across all of 1.0–3.5.)
- **Skip changes too small to matter to the player.** If a fix only benefits the dev/code, leave it out.
- **Per-version entry caps (independent per version):**
  - A major `.0` release holds **up to 20 entries/lines**.
  - Each minor release after it (`.1`, `.2`, …) holds **up to 10 entries/lines**.
  - When the in-progress version is full, **roll to the next minor** rather than overflowing.
- **Header format:** each version block starts with a line `New in X.Y.` (trailing period), matching the existing file.
- **Order is reverse-chronological at both levels:** the newest version block goes at the TOP of the file, and within a block the newest entry goes at the top.
- **Version number — `build/version.txt` is the single source of truth, but the DEV owns it: do NOT edit it yourself.** See [[dont-touch-version-txt]]. When opening a new `New in X.Y.` block, **flag that the version needs bumping and let the dev set it** — never write `version.txt` unless they explicitly ask (they sometimes set it by hand for unrelated reasons, e.g. matching another server's version to test a connection). The two per-side constants `src/client/includes/version.nvgt` and `src/server/includes/version.nvgt` (`const string version="X.Y";`, included from each `includes.nvgt` before the globs) are **generated** from `version.txt` — both launchers (`cfc.py`/`cfs.py`) copy it in before each run, and `build/tools.py` does the same at compile — so never hand-edit those either (that generation replaced the old 2026-06 "bump both constants by hand" rule).

- **A large, multi-section feature spans MULTIPLE entries across MULTIPLE versions — ship it in stages, one player-facing entry per chunk as it lands.** The per-version caps enforce this. Example: the PVP mode spread across 3.1 (arena, side selection, phases, swinging-no-damage) and 3.2 (real wall damage, defender reinforce, winner/results screen, saved win-loss, forfeit countdown, through-the-wall audio). The planned weapon/item data-driven reorg (tasks #2/#3/#5, the 4.0 flagship) is the next such feature and will accumulate entries the same way. IMPORTANT: the internal/invisible parts get NO entry — the data-driven loader + value migration (#2), like the earlier GameEngine64.dll removal, change nothing the player can feel; only the player-facing features earn lines (#3's weapon-selection menu, #5's new ranged weapons + gathered ammo + reload + the firebomb wall-fire, possibly the four wood varieties).

**Why:** These rules keep the changelog readable and scoped — per-version caps prevent bloat, the sentence limit prevents padding, and the version-bump rule keeps the in-game/server version strings consistent with what shipped.

**How to apply:** Any time a changelog entry is written or a new version block opened, follow these rules exactly; the caps are hard limits. Related: [[confirm-before-implementing]], [[list-modified-files]].
