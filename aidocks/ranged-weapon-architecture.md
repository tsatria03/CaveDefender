Architecture for the ranged weapon system, now BUILT (server `globals/bullet.nvgt`, plus client/server ammo + reload). Ported in SPIRIT from SimpleFighter's bullet.nvgt / weapon.nvgt, but re-architected because SF's combat is offline/local and CaveDefender is server-authoritative multiplayer. The tuning numbers live in the data files ([[game-data-file-layout]]).

**Targets = WALLS ONLY.** This is NOT an FPS. Ranged weapons (crossbow, cannon, firebomb, grenade) fire at the four cavern walls; players are NEVER bullet targets and CANNOT be hit by a moving bullet even while walking/running through its path -- a bullet passes harmlessly through players. Hitting players would defeat the purpose (Wallbreaker: the walls are the objective). So bullet collision checks walls and nothing else.

**Who can use what (SETTLED 2026-07-15).** The 4 MELEE weapons (axe/bat/crowbar/hammer) can be used by BOTS (PVE enemy bots swing them, via enemy_damage/weapon_id_for_name) as well as human players. The RANGED weapons (crossbow/cannon/firebomb/grenade) are HUMAN-ONLY -- bots never use them; only human players wield them, in EVP and PVP (the attacking side). So the ranged/ammo/bullet system is a player-only subsystem.

**Server-authoritative, mirroring the existing melee model.** The client's melee swingloop() already does: on held Space, windup -> send "attack <weapon>" -> server picks nearest wall in reach, applies damage, broadcasts "wallhit"; client plays the swing locally, server broadcasts the impact so all players/watchers hear it. RANGED is the same loop plus a traveling bullet. The client sends fire/reload intents; the SERVER owns ammo, the bullet, damage, and wall-fire. The client plays audio only (draw/fire/empty/reload/ping on the actor; the travel loop and impact at the bullet's position) -- same client/server split walls already use.

**Bullet sync = OPTION A-PLUS (resolve-on-fire, hold for flight, impact on arrival).** On fire the server ray-marches from the shooter along their facing to the first wall within range, computing the target wall AND the arrival time (fire-time + range/speed). It HOLDS a lightweight bullet entity `{origin, dir, target_wall, impact_tick}` and applies the damage + broadcasts the impact only when the flight timer elapses. Clients play the travel `loop` for the flight duration and the impact at the end. ONE "bullet spawned" message + ONE "impact" message per shot -- NO per-cell streaming. This keeps the impact sound honestly timed with the travel audio (plain resolve-on-fire would land wall damage/sound before the arrow finishes flying). Since walls are static, locking the target at fire-time loses nothing. Streaming positions cell-by-cell (the faithful-but-heavy alternative) was rejected: only needed for moving targets, which don't exist here. The server bullet is therefore FAR leaner than SF's per-cell bulletloop (which collides vs npcs/doors/vehicles/spikes/etc -- none exist in CaveDefender).

**Ammo + reload (server-side, ported from SF weapon.nvgt).** Model: loaded_ammo (magazine) + a reserve. fire(): if loaded_ammo==0 -> client plays `empty` (dry-fire), no shot; else decrement and spawn the bullet. Reload uses SF's SOUND-LENGTH TIMER (SF start_reload(): `reload_duration = p.get_sound_length(sound_slot); reload_timer.restart(); is_reloading = true;`; check_reload() completes when `reload_timer.elapsed >= reload_duration`, moving reserve -> magazine). PING RESTORED (user, 2026-07-16, reversing the original no-ping call): a `ping.ogg` now plays when the reload timer elapses (reloadloop, weapon.nvgt) because the reload clip alone was too hard to time -- played 3D at the player AND relayed via "step weapons/<class>/<name>/ping" so others hear the ready-ping too. The dev has ping.ogg for all 4 ranged weapons. So the ranged weapon sound set is now draw/empty/fire/hit/loop/reload/rico/ping. Ranged weapons use SINGLE fire/hit sounds (no melee 1-3 variants) plus empty/reload/loop -- see [[audio-and-sounds]] for the per-class sound-folder layout.
Reload timing model (SETTLED 2026-07-15 = option B, no data field): the CLIENT measures the reload sound with `get_sound_length` (confirmed in client sound_pool.nvgt:517) and drives the reload timer itself -- exactly like swingloop() already times the melee wind-up before telling the server -- then notifies the server on completion; the server validates the reserve and moves ammo (reserve -> magazine). There is NO `reload` duration field in the weapon file: the sound IS the timing, so it self-syncs (swap the reload clip and the reload gets faster/slower automatically). The server stays authoritative over the AMMO move; it only cedes the exact millisecond of the delay, which isn't balance- or security-sensitive for this co-op wall-breaking game.

**Ammo type per weapon (SETTLED 2026-07-14).** Each weapon has its own ammo type (names taken from SF's allowed list: normal, gas, air, battery, powder, arrow, rocket, water, knife, needle, dart, ball, mana, fuel, rock, grenade). No shared pools:
- crossbow -> `arrow`
- cannon -> `powder`
- firebomb -> `gas`
- grenade -> `grenade`
Crossbow and cannon are LAUNCHERS (feed on a separate munition, a normal magazine). Firebomb and grenade are THROWN CONSUMABLES -- the weapon IS its ammo, so magazine size = 1 (hold one, throw it, "reload" pulls the next from your reserve/stock). Same ammo state machine for all four; only magazine size + ammo type differ. Maps straight onto the `ammo type` field in the weapon data files.

**Weapon data-file format (task #2).** SF's per-weapon info.sif is the template: key=value lines (damage, x/y/z range, bullet speed, repeat time, spam time, ammo, loaded ammo, max ammo, launch type = bullet|projectile|bomb:subtype, ammo type, ...). Our .mel/.exp/.arc files should follow this shape; a server loader parses them at startup.

**Firebomb tie-in.** Firebomb is an explosive that travels (loop) then, on wall impact, triggers the wall FIRE mechanic (burnstart -> burning loop -> burnstop, a burn-over-time on wallwood) on TOP of its impact damage. Burn config comes from the weapon's `status=burn:duration:tick:damage` field (see [[game-data-file-layout]]).

**Burn stacking = INTENSIFY (SETTLED 2026-07-15, modeled on real fire).** A wall has ONE burn state (intensity + remaining time). Impact damage always lands on every throw. If the wall is ALREADY burning when another firebomb hits, the burn does NOT reset-to-same, get ignored, or spawn a parallel burn -- it INTENSIFIES: the damage-per-tick goes UP (hotter, e.g. add the bomb's burn damage on top of the current rate) AND the duration refreshes. So repeated firebombs make the fire hotter and consume the wall faster (pouring fuel on a fire). UNCAPPED -- the throttle is the ammo economy (gas is gathered, magazine=1, reload between throws), so it can't be spammed; a coordinated team stacking heat is a legit strategy, not an exploit. (A hard intensity cap, e.g. 3x, is an option if it ever proves too strong.)

**Burn AUDIO -- exactly ONE burning loop per wall, never layered.** The client holds a single `burning` loop handle per burning wall for the whole burn. On a re-hit of an already-burning wall it must NOT start a second loop (that would stack into loud noise) -- it keeps the one loop playing and lets the firebomb's own impact sound give the hit feedback. `burnstart` plays on EVERY firebomb hit -- the initial ignite AND each re-hit of an already-burning wall -- as the "fresh fuel / fire intensified" cue, so players hear that they've stoked it hotter; it's a SHORT one-shot so it doesn't pile up the way a loop would. Only the `burning` LOOP stays single (never restarted/layered). `burnstop` plays once when the burn finally lapses and the single loop is stopped. So any number of firebombs on one wall = one loop (never louder) + a short burnstart whoosh per hit (the intensify feedback).

**Ammo source = GATHERED off the ground, exactly like wood (SETTLED 2026-07-14).** Ammo is a pickup resource mirroring the defenders' wood economy: just as wood drops to gather, ammo drops in the four types to gather. Sounds live under `cf/client/sounds/objects/items/ammo/<type>/` (arrow, gas, grenade, powder) as siblings of `items/wood/<type>/`; server data files are `cf/server/data/objects/items/ammo/<type>.amt` (see [[game-data-file-layout]] for the full data tree + extensions). Ammo uses a SUBSET of wood's item sound set: `drop`, `loop`, `take` -- but NO `place`. Wood has `place` because you place wood INTO a wall to reinforce; ammo is never placed in the world -- taking it goes straight into your reserve, and the "load it into the weapon" step is the WEAPON's `reload` sound (not an item sound; and there is NO ping). So the item layer = drop/loop/take, the weapon layer = the reload. (The ammo sound folders exist but are EMPTY -- the dev still needs to source those three sounds per type; note `powder` is currently missing even `drop`.) Server-side, the existing item system (globals/item.nvgt: item class with a `type` string, is_wood(), spawn/pickup) extends to carry ammo types alongside the wood types; picking one up adds to that ammo type's reserve for the matching weapon. NOTE: the type is spelled `gas` (one s); the folder was renamed from `gass` -> `gas` so the sound path, data file `ammo type`, and wire messages all match.
