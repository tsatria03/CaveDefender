**EVP (Environment versus Player)** is a game mode — the reverse of the existing PVE [[game-vision-wallbreaker]] mode. BUILT and feature-complete; this doubles as the as-built reference. Listed BEFORE PVE in the "Choose a game mode to play." menu (the mode-selection step added ahead of the who-can-play/size steps).

**Role inversion vs PVE:** the four walls are the bots' to defend; players attack them. Players wield weapons and smash walls; bots gather wood and repair walls (bots are PURELY builders — they never attack players in v1). Players attack from OUTSIDE the cave, damage registers inside.

**Builder bots (implemented — server `globals/builder.nvgt`):** the mirror of the attacker `enemy`, but they PHYSICALLY MOVE. Each has `x/y`, a `movespeed` (rolled once at spawn, random 100-200ms, kept for life), a `movetimer`, and a small state machine: SEEKING (walk toward the nearest ground wood via `nearest_wood_index`; on its tile, take it — `removeitem` + positional `footstep .../wood/take` — and pick the weakest standing wall) → CARRYING (walk toward that wall; within `wall_reach` (5) tiles, pause one beat = `movespeed`) → PLACING (`repair_wall(wi, random(8,16))` + `footstep .../wood/place` at the bot, back to seeking). Each step is orthogonal (`builder_step_toward`, x then y) and emits a `footstep objects/platforms/caveground/stepN` at the bot's spot (so bots-inside sound distinct from players' gravel steps). Idle/silent when no wood or no standing wall. `builder_tick()` runs from `gametick` beside `enemy_attacks()`. Helpers: `spawn_builders(gameid,count)` (scattered random interior tiles, each rolls its own speed), `clear_builders` (everywhere `clear_enemies` is), `builders_in`, `weakest_standing_wall`, `wall_distance`, `nearest_wood_index`. `movetimer` pauses/resumes with the game. `evp_builder_count()` = 3 (base; escalation is #64). **Player wood pickup is disabled in EVP** (`gather_wood` early-returns) so the bots get the wood. The take/place sounds ride the existing client `footstep` handler. PASS 2 (#68, implemented): bot positions are synced to clients for the B locator. Server messages: `botspawn <id> <x> <y>` (on spawn, and `send_builders_to` resends all to a player joining/spectating mid-round), `botstep <id> <x> <y>` (each move — REPLACES the generic footstep for bot moves: the client plays the caveground step AND updates the roster), `botclear` (drop all, sent by `clear_builders`). Client roster in `globals/bot.nvgt` (`clientbot` list + `set_clientbot`/`clear_clientbots`, cleared in `draw_map`). The B key (EVP only, players + watchers) runs `track_bots()` in mapfuncts — lists ALL builders nearest-first like track_players: "Builder is <dir relative to facing>, about N tiles away."

**Round structure (endless, mirrored — "Option B"):** score = rounds cleared. A round is cleared when all four walls are brought to 0; then walls rebuild tougher and bots repair faster / more builder bots spawn. Crucial: the attacker count is FIXED by who joined (1 solo, up to 4 open) — you can't spawn more humans — so ALL round-to-round escalation comes from the DEFENDER side (tougher rebuilds, faster repair, more builders), unlike PVE where escalation = more attacker bots each wave.

**Map:** outdoor gravel exterior. APPROACH DECIDED: `draw_map` gets an `outside` bool param (NOT new mapnames — the existing `"cavern"`/`"cavern2"` keys stay). When `outside` is true the same branch lays `gravel` instead of `caveground`, `cavern_ext` instead of `cavern_int`, and (for the deck) `wallwoodpile` instead of `wallwoodcrate`. Walls stay `wallwood` (no special outside wall); floor = the `gravel` platform (sounds present: objects/platforms/gravel/step1-5.ogg, same naming as `caveground`). Gravel chosen over plain dirt for audio contrast with the rocky `caveground` interior. Players roam the gravel outside and attack the wall border. Outdoor ambience bed = `cavern_ext.ogg` (the interior cavern bed was renamed `cavern.ogg` -> `cavern_int.ogg`); music stays `cavern` (only ambience was split int/ext).

**Wall strike volume:** the PVE danger cue `wall_strike_volume()` (client wall.nvgt — quiet at full strength, loud as the wall dies) is DISABLED when attacking from outside; strikes play at constant volume, gated by the inside/outside flag. Wall health is read via the n/e/s/w/t readouts instead.

**Watchers (implemented):** the spectator overhead deck (`cavern2` in client `draw_map`) is `outside`-aware — EVP draws a `gravel` floor, `cavern_ext` ambience, and a `wallwoodpile` indestructible border; PVE draws `caveground` + `cavern_int` + `wallcrate`. The spectate handler passes `game_outside` to the `cavern2` draw, so watching an EVP game gives the outdoor deck. Same overhead-deck mechanism otherwise (z = spectate_height); a watcher's own footsteps aren't relayed to the players below.

**Weapons:** players are handed all four on the number row — KEY layout 1=axe, 2=bat, 3=crowbar, 4=hammer. Selecting plays that weapon's `draw.ogg` and posts "x equipped" to the misc buffer (which speaks AND logs). **You are UNARMED until a round starts:** the client arms you with the axe when `gamephase` becomes build or wave (covers round start and a mid-round join; `equip_weapon("axe")`) and disarms you (`current_weapon=""`) on idle/over. The number keys only switch weapons while armed; swinging requires a weapon, so there's no attacking before the round. NOTE: this key layout differs from the internal bot weapon-id scheme (1=axe, 2=crowbar, 3=hammer, 4=bat, see client wall.nvgt ~line 107) — keep "key number" and "internal weapon id" as a clean, deliberate mapping so they don't get confused.

**Attack mechanic (windup):** hold Space → swing sound (`fire1-3.ogg`) plays immediately → after a per-weapon WINDUP (axe 300ms … hammer 600ms) the hit lands: damage applied + `hit1-3.ogg`, then a per-weapon RECOVERY cooldown before the next swing. The swing does NOT lock the player — you keep moving and can switch weapons freely (modeled on SimpleFighter's `is_swinging`, which only blocks starting another swing). The in-flight swing captures its weapon at the start (`swing_weapon`) so a mid-swing switch doesn't change the pending hit. Heavier weapon = longer windup + bigger damage (reuses the bots' per-weapon `enemy_damage`).

**Phases (implemented):** idle → build (bots reinforce the walls; players CANNOT damage them yet, swings ricochet) → wave (players attack continuously, bots repair continuously). NO attack/rest alternation like PVE — once the build phase ends, both sides act nonstop. `set_game_phase` forks on `mode=="evp"`: build spawns the builders (`spawn_builders(evp_builder_count(), evp_builder_interval())`, base 3 @ 2000ms) and skips the player wood grant; wave skips the attacker setup (no `fill_walls_to`, no `wave_cycle`). `gametick` skips the PVE wave-warnings, attack/rest flip, and the all-walls-down `end_round_loss` for EVP (`mode!="evp"` guards). Wood still drops (builders consume it). Client phase announcements + the Tab status are `game_outside`-aware ("the bots are reinforcing" / "tear down the walls!"). EVP players' wood keys are inert: reinforce (Shift+Enter), carry-count (I), locate-wood (D) all gated off with `!game_outside`. STILL TODO (#64): all-walls-down = round CLEARED → score (rounds cleared) → escalation (tougher walls / more/faster builders) → next round, plus the loss conditions (round timer + stall). Until then an EVP round plays build→wave but doesn't resolve when you clear the walls. **Build-phase feedback (implemented):** a player's swing that's in reach of a wall during build RICOCHETS — the server broadcasts `wallrico <dir> <weaponid>` to the whole game (via `send_to_game`) and clients play the weapon's dedicated `rico.ogg` (no wall creak, no damage) via `play_wall_rico` at the wall, so you hear you're on a wall before hits count. Each weapon has its own `weapons/<w>/rico.ogg` alongside draw/fire1-3/hit1-3. During the wave the swing damages and broadcasts `wallhit` to everyone. (This ricochet replaces the originally-planned "You can't attack while the bots are reinforcing." text message — audio feedback instead of a nag.)

**Weapon sounds are all 3D and heard by other players/watchers:** the DRAW and the FIRE (swing) play locally via `p.play_3d` at your own spot AND are relayed positionally at the swinger's spot via the existing `step`→`footstep` relay (so others hear `weapons/<w>/draw` and `weapons/<w>/fire<n>` in 3D where you stand); the HIT (`wallhit`, real damage in the wave) and the RICOCHET (`wallrico`, build phase) are broadcast at the WALL via `send_to_game` (3D via `play_wall_hit`/`play_wall_rico`). So: draw/fire emit from the player, hit/ricochet from the wall. SLOTS: only the DRAW uses `wepslot` (re-drawing cuts the prior draw); fire and hit are untracked so they overlap and a new swing never destroys a still-ringing hit.

**Win/loss + scoring ("Option 4", IMPLEMENTED — server `game.nvgt`):** the EVP round resolution runs in `gametick` (a pass after the PVE one, guarded `mode=="evp" && phase=="wave"`):
- **Win (clear):** all four walls down → `evp_clear_round()` — bumps `evp_rounds_cleared` (the score), sends `evpcleared <n>` (client announces "Round N cleared!"), clears builders/items, rebuilds fresh walls (`gamereset`), and `set_game_phase(build)` for the next, tougher round. AUTO-continues (no results screen, no host re-begin). Players keep their weapon across rounds.
- **Escalation:** TWO levers. (1) More builders: `evp_builders_for_round(rounds_cleared)` = 3 + cleared, capped at 8. (2) **Wall toughness (added 4.1):** each wave the standing walls are guaranteed to at least `maxhealth * evp_fortify_mult(rounds_cleared)` where `evp_fortify_mult = evp_fortify_base + cleared*evp_fortify_step` (defaults 1.3 + 0.3/round → round 1 130%, round 2 160%, …), clamped ≥ 1.0. Enforced at EVP wave start in `set_game_phase` (top-up of health AND floor per standing wall, breach skipped) so it's deterministic, not left to wood luck; the builders' own repair cap is this same mult (`repair_wall(..., evp_fortify_mult(cleared))`), so bots can build past the old fixed `wall_overbuild_cap`=4.0. Tunable via conf.gvr `evpfortifybase`/`evpfortifystep` (loader in gamedata.nvgt). BEFORE 4.1 the only lever was builder count, which doesn't lift the wall peak (wood-supply/time-bound), so walls plateaued ~130% every round — that was the bug. maxhealth itself stays constant (so the wall % readout, health/maxhealth, is what visibly climbs). Round timer stays constant per size.
- **Builder stale-target fix (4.1):** if a bot's target wall is felled during its 1-beat placing pause, the place beat no longer mimes placing onto the breach / wastes the wood — it re-aims at the weakest wall still standing and keeps carrying (builder.nvgt place-beat; the carrying state already re-picked, this closes the placing-state gap). Was task #18.
- **Round timer:** `evp_round_secs` = `evp_round_seconds(size)` (base per map size) + `evp_rounds_cleared * evp_round_bonus`, set at wave start; `wavetimer` counts against it but ONLY advances during attack segments (pauses in rest windows — a budget of real attacking seconds, not wall-clock). Sent as the wave's `gamephase` secs so the client Tab status shows "N seconds left." **HOST-TUNABLE (4.2, EVP-only):** base is a size-tier lookup `tier_value(evp_round_sizes, evp_round_secs_vals, size)` seeded to 75/120/180/180s at sizes 25/50/75/100, and `evp_round_bonus` (default 10) is the per-cleared-round bonus — both in gamedata.nvgt, loaded from conf.gvr `evproundtime_<label>=<size>:<secs>` / `evproundbonus`. PVE has no survival clock; PVP uses its own `pvp_round_seconds`, so these knobs are EVP-only. STILL hardcoded: the attack/rest segment length (`buildduration/2`, shared with PVE/PVP) and `build_seconds` (45/75/120, shared).
- **Stall watchdog:** `evp_stalltimer` + `evp_lowest_hp`. Each tick, `total_wall_hp` is compared; a new low = progress (update lowest, restart stalltimer). Window = `evp_stall_window_ms` = round_secs/3 (always shorter than the round, scales per size). Pauses/resumes with the game.
- **Loss:** round timer expires with a wall standing OR no new low for the stall window → `end_evp_loss(gi, "time"|"stall")` → phase "over", `evpover <cleared> <reason>` → client `handle_evp_over` (results + play-again, reusing the PVE playagain flow; `reset_for_new_round` zeroes `evp_rounds_cleared`).
