**The custom game engine.** Map objects are spawned by a custom engine written in **PureBasic**, shipped as `cf/client/lib/GameEngine64.dll` and wrapped in `src/client/includes/main/deps/GameEngine.nvgt` (`load_engine()`, `spawn_platform` / `spawn_zone` / `get_tile_at`, plus FTP/zip/download/recording helpers that are part of the engine's surface and not all used yet). The wrapper loads it cwd-relative as `engine.load("lib/GameEngine64.dll")`, which resolves to `cf/client/lib/` because the launcher's cwd is `cf/client` (and to the bundled `lib/` next to the exe in a compiled build). The DLL must be present, or the wrapped calls throw `"library is not loaded"`; `load_engine()` checks the load result and shows a clean alert + exits if it fails. (The voice-chat encoder `oggenc2.exe` sits beside it in `cf/client/lib/`, invoked the same cwd-relative way. This DLL is deliberately kept as a real file even when boxing — see [[enigma-boxing]].)

**Why it's being migrated away (the destroy limitation).** The DLL's only tile-removal call is `clear_map()`, which wipes the ENTIRE map at once -- there is NO per-instance removal (no `remove_platform`/`destroy_platform`/`update_platform`), and `spawn_platform` returns `void`, so you never get a handle back to a specific tile. This is the real reason the DLL was a bad fit for walls and platforms: you cannot destroy or move ONE of them. The game works around it by PAINTING OVER: client `wall.nvgt` `destroy_wall_client()` doesn't remove a broken wall, it `spawn_platform`s a floor tile on top and leans on latest-wins lookup, leaving dead tiles stacked underneath until the next `clear_map()`. The fix is to go native (SimpleFighter's `platform.nvgt` model): `spawn_platform` returns an int index handle, `remove_platform(id)` sets `@platforms[id]=null`, lookups skip nulls -- real per-instance destroy, no DLL, no pointer marshaling. The live client surface is tiny -- only `clear_map`/`spawn_platform`(2D)/`get_tile_at`, plus `load_engine()` at startup whose return is unused; ALL the wrapper's other functions (zones, import/export, recording, download, FTP, zip, file dialog, regex, email) are defined but never called. The SERVER is already fully native (its `globals/wall.nvgt` uses no DLL), so the migration is client-only. See [[include-tree]] for where the wrapper lives.

**Non-stock NVGT engine change.** The game depends on a **patched build** of NVGT (the Legacy-NVGT C++ source — a separate repo, not this game's `src/`; see [[nvgt-engine-source-location]]). `library::load()` in the engine's `src/library.cpp` was changed to resolve **any** relative path against the current working directory (not just bare filenames) before loading, so a DLL in a subfolder like `lib/GameEngine64.dll` loads correctly — Windows' `SafeDllSearchMode` otherwise excludes the CWD. This is what lets `engine.load("lib/GameEngine64.dll")` find the DLL under the launcher's cwd (`cf/client/lib/`). It needs a `scons` rebuild of NVGT — already baked into the pinned `C:\nvgt2` fork (see [[engine-pinned-to-nvgt2]]). With **stock** NVGT the engine won't load from `lib/` at all.
