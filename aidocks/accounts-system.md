Server-side. Each account is a **folder** at `cf/server/data/players/<username>/` (resolved cwd-relative as `data/players/<username>/` while the server runs; the folder **keeps the case the account was created with** — `DevilGuardian`, not `devilguardian`; `account_dir` resolves it case-insensitively so login still works regardless of case) holding one small file per field, so a server host can read or edit a single value by hand: `username.usr`, `email.usr` (a **cosmetic, unverified** email collected at account creation — the new-account form is username/email/password/confirm-password, the email must be non-empty and contain `@`, no spaces, confirm-password is a client-side match check only; stored as-is and hand-editable, **never used for login** — but **required to exist**: `login()` refuses any account whose `email.usr` is empty/missing, so legacy accounts made before the email update must be recreated; there is no verification e-mail, though NVGT's built-in HTTP client could send one via an email API if that's ever wanted), `created.usr` (the account creation date as a human string like `July 3, 2026`, written once at registration from the engine's `DATE_*` globals; shown as "Member since" on the F6 card, `Unknown` for accounts predating the field), `nickname.usr` (the display name shown in place of the username **everywhere a name is spoken**; auto-created equal to the username, hand-editable, reverted with `/nick default`), `password.usr` (holds `argon2id$<nb_blocks>$<nb_passes>$<salt>$<hash>` — a random CSPRNG salt and the memory-hard **Argon2id** hash of the password, cost params inline so a verify reproduces them; the salt is not secret, and there is **no SHA-256 fallback**, so any pre-Argon2 account must be re-registered), `pronoun.usr` (how third-person messages refer to the player: the bare key from `data/preffs/pronouns.svr`, e.g. `Masculine/male` / `Singular/they` / `None/unknown` — the default is the configured they/them option, a `(name only)` entry uses the name, set in-game with `/pronoun`), `rank.usr`, `muted.usr`, `banned.usr`, and the saved-room files `pubroom.usr` (`1` = has a public room, blank = none) and `privroom.usr` (the private room's password, **stored encrypted** as a single `<ciphertext>:<key>` blob mirroring the old account `<salt>:<hash>` layout — a fresh random per-set key (CSPRNG via `make_salt`) plus the AES-encrypted, base64-wrapped password, joined by a colon, so it's reversible and Retrieve password still works; this is **obfuscation**, the key riding beside the ciphertext keeps the password out of plain sight but isn't safe from someone who has the file — blank = none). Saved rooms are **recreated when the host logs in** (and close on disconnect, but the files keep them for next time); a server host can hand-create `pubroom.usr` to pre-make a public room, but `privroom.usr` is encrypted, so set a private room's password from in-game instead. `muted.usr`/`banned.usr` each use one scheme: **empty = not muted/banned, `0` = indefinite/permanent, any other number = the absolute epoch it expires at** (so the old separate `muted`/`muteuntil` pair is now a single file). `banned.usr` additionally may carry an **optional ban reason colon-embedded after the time** — `<until>` or `<until>:<reason>` (e.g. `0:spamming` or `1783538914:being rude`); `parse_epoch` reads only the leading digits and stops at the colon so `get_ban_until` is unaffected, and `get_ban_reason` returns everything after the first colon (reasons may themselves contain colons). Usernames can never contain a colon (filterchar strict mode), so the delimiter is safe. Field access goes through `read_field`/`write_field`; `account_exists` checks the folder; `/staff` enumerates accounts via `find_directories("data/players/*")`. A new account writes only the fields that hold a value (see the 3.5 changelog); blank files (pubroom/privroom/muted/banned/etc.) are created on first use.

`rank` is one of `player`, `moderator`, `administrator`, or `developer` (mutually exclusive, **full words only** — no short forms). `rank_level()` maps them to `0/1/2/3` and **commands are gated by minimum level** (`/promote`/`/demote`/`/kick`/`/motd`/`/warnings`/`/mutes`/`/bans` ≥ moderator; `/ban`/`/tempban`/`/unban`/`/mute`/`/unmute`/`/warn`/`/changename`/`/changepass`(admin)/`/delchar`(admin)/`/restart`/`/fastrestart` ≥ administrator). Roughly: **moderators view, administrators act**. `is_staff` (= level ≥ 1) is for **display only** (chat-name prefix, `/staff` listing, the "you are now…" notice). `/promote <name> <rank>` and `/demote <name> <rank>` both take an explicit target rank: you can act on anyone **at or below** your own level (yourself and peers included) but never someone above you, and a **grant cap** means you can never set a rank above your own (the anti-escalation guard). Each also enforces **direction**: `/promote`'s new rank must be *above* the target's current one, `/demote`'s must be *below* it — so the verb always matches and you can't accidentally over-drop someone. Helpers live in `src/server/includes/main/globals/account.nvgt` (`account_exists`, `create_account`, `load_account`, `read_field`, `write_field`, `verify_password`, `get_rank`, `set_rank`, `rank_level`, `is_staff`, per-channel mute + ban getters/setters, `now_epoch`, `rename_account`, `delete_account`, plus the owner-alias helpers).

`/mute <name> [seconds]` (0/omitted = indefinite) silences a player's chat and voice — staff can mute **players only**, never each other; mutes persist in the account file (surviving reconnects and restarts) and timed ones auto-expire (polled in `netloop`). Banning is **split into two commands** (both **administrator+**, **at-or-below target protection**, self-ban allowed): **`/ban <name> [reason]`** = permanent, **`/tempban <name> <seconds> [reason]`** = timed (seconds **required, ≥ 1**). Both take an optional free-text reason after the required args, stored colon-embedded in `banned.usr`, shown to the banned player (login-rejection message and the online "banned" push) and in the staff `/bans` viewer, but **kept out of the public ban announcement**. A ban is enforced both in `login()` and by kicking the target off immediately; persists the same way, and timed bans are cleared at the next login attempt. Shared execution helper `perform_ban(actorpeer, ti, dur, reason)` in server net.nvgt; the F6 card ban flow walks length → "provide a reason?" → reason and routes to `/ban`/`/tempban`. (The old single `/ban <name> [seconds]` form is gone; the card's dead Ban-status line and unban button were removed since a banned player is never in the roster.) `/kick` is **moderator+** with **at-or-below-rank protection** (players and staff at or below your rank, including yourself/peers, never someone above you; temporary — the kicked player can sign back in). `/kick` and `/ban` may both target staff (at-or-below), unlike mute (players-only). `now_epoch()` returns whole Unix seconds via NVGT's Poco-backed `timestamp`. Passwords are **never** stored in plaintext.

**Offline private messages:** `/pm` is offline-aware — an online target gets the live PM unchanged, but an offline target whose **account exists** (resolved by username; nicknames only work for online players) has the message **queued** rather than erroring. Queue lives in `offline_pm's.usr` (the apostrophe is **intentional, not a typo** — NVDA reads "pms" as a garbled word but speaks "pm's" cleanly as "P M's" when a host browses the account folder by ear; don't "correct" it back), one message per line, **TAB-delimited** `<sender username>\t<message>` (tabs because the message is free text; unlimited, nothing dropped). `add_offline_pm`/`get_offline_pms`/`clear_offline_pms` in account.nvgt; `account_ignores(recipient, sender)` is the **offline-capable** `player_ignores` (reads `ignored.usr` from disk, staff exempt) used to bounce a queued PM the recipient has muted. Delivered on **fresh login only** (not reconnect) by `deliver_offline_pms(index)` in net.nvgt, which sends each as a channel-0 `offlinepm <sender>\n<display line>` control message (client shows "Offline PM from <name>: <msg>" in the private chats buffer with the PM sound and sets `last_pm_sender` to the sender USERNAME so `/reply` reaches them even while offline) then clears the file. Sender gets an "Offline PM to <name>" echo.

The on-disk rank loads into the in-memory `player` roster at spawn; there is **no** blanket `dev` backdoor (uncompiled builds used to auto-elevate — removed). Instead the configurable **owner** (`conf.svr` `owners=`, a **comma-separated list of up to 3 case-insensitive aliases for the one owner** — the same person's alternate account names — default `tsatria03`; the legacy singular `owner=` field is still read as a fallback) is always forced to `developer`, both on account creation and at login (`is_owner` = membership in the `owners` list, in account.nvgt), and **cannot be demoted** (the `/demote` handler refuses any owner alias). When an owner-alias account is **renamed** its alias is updated (`replace_owner_alias`); when **deleted** just that alias is dropped (`remove_owner_alias`); both persist via `writeprefs()`.

**Nicknames** (see [[nicknames]]): the **username is the identity** (login, folder, all command targeting); the **nickname is the display** shown in its place everywhere a name is spoken (chat, `/who`, `/staff`, the R/L/G lists, the P tracker, lifecycle/AFK notices). `get_nickname`/`set_nickname` back `nickname.usr`; the roster carries `players[].nickname` and ALL `display_name*` helpers return it instead of `.name`. `/nickname`/`/nick <name>` sets your own (or `default`); `/nick <username> <altname>` lets **any staff** set an online player's; `apply_nickname` validates (username rules; unique among online nicknames AND usernames, and not equal to **any registered account's username** other than your own), persists, and alerts everyone.

**Command targeting:** `get_player_index_from` resolves a typed name to an ONLINE player by **username first, then nickname** (both **case-insensitive**), so commands accept either. Player-targeting commands require the target **online** and operate on `players[ti]` — the **only** offline-capable holdout is **`/unban`** (a banned player is always offline), which takes the username resolved case-insensitively. Not-found feedback is context-aware: "This player does not exist." vs "This player is not online." `/who` and `/staff` append `(username)` when it differs from the nickname.

Protocol edges: `register <username> <email> <password>` creates an account (all three tokens space-free); `login <username> <password> <version>` verifies it — the server **rejects the login if `<version>` doesn't match its own** (client and server must be same version — the mismatch message is direction-aware) and **rejects it if the account has no email on file**. The client **connection menu** (`menus/menu.nvgt`) offers **sign in** (last-used, cached in the encrypted `settings.cvf`), **set up an existing account**, and **new account**. **Pronouns** (`/pronoun`) mirror the language-channel system: `data/preffs/pronouns.svr` lists options (`key (subject/object/possessive/possessive-pronoun/reflexive)`, or `none (name only)`), read fresh; `/pronoun` pushes a server menu, the choice persists to `pronoun.usr` and loads into `players[].pronoun` at spawn (validated, falls back to `default_pronoun()`). `pronoun_word(key, form, displayname)` returns the right word for a message. The **rules agreement** gate is server-side too — see the [[enigma-boxing]]-adjacent rules-version/`ruleswitch` design in the changelog.
